# AI生码

AI生码（AI辅助代码生成）是淘天营销中后台在财年初推进的AI提效核心实践，旨在通过AI技术覆盖需求交付全链路，将人工流程串联成本降至最低。

## 概述

**来源文档**：《淘天营销中后台生码工作流最佳实践》（淘天集团-营销前台技术团队）

**核心目标**：统一收敛AI生码路径，扩大可落地场景范围，提升AI生码实际效果。

## 核心概念

### 云端托管生码

基于 **AoneSuper**（集团内提供云端生码沙箱环境的平台）实现统一收敛，替代本地研发模式。

**为什么不选本地**：
- 环境配置难以统一，问题排查协作成本高
- 生态用工的 AK 管理困难
- 执行易中断（依赖电脑持续在线）

**云端一体化研发流程**：
- 云端生码：绝大多数场景可云端执行全流程
- 本地IDE直连沙箱：少数复杂场景需人工干预时使用

### 跨仓库工作区

支持多仓协同的工程能力，基于 **git submodule + turborepo** 构建。

**设计思路**：
- 聚合需求相关的所有仓库到同一工作区
- git submodule 提供 git 托管能力
- turborepo 实现项目间依赖的自动 link 配置

**核心能力**：
- 消除多仓库来回切换
- 子仓库代码修改提交到对应git空间
- 一键启动所有需求子仓库的服务

### 场景化工作流

可编排的工作流框架，覆盖需求理解→编码→构建发布全链路。

**整体架构**：
- **固定节点**：需求准备（入口）、构建部署（出口）
- **动态节点**（1~N个）：由需求场景定义，每个节点绑定独立 Command

**节点机制**：
- Command 本质是一份 Prompt，定义节点任务和行为
- 可声明调用 MCP（外部工具）和 Skills（内置技能模块）
- 节点顺序执行，逐步推进

### 差异化生码策略

根据任务确定性不同，采用不同优化策略：

| 类型 | 场景 | 策略 |
|------|------|------|
| 高确定性 | 迁移/重构 | 架构说明文档 + 领域Skill固化规则 |
| 低确定性 | 日常迭代 | 功能树实现查表式知识供给 |

#### 迁移/重构（高确定性）

**核心思路**：规则前置固化 + 验证系统化

**关键机制**：
- **架构说明文档**：新旧架构对照、禁止行为清单、接口映射规则
- **领域Skill**：ProForm Skill（表单开发）、ProTable Skill（表格开发）
- **工作流节点化**：了解旧架构 → 计划制定 → 编码实现 → 验证收尾

**落地效果**：
- 商家营销工具N合一迁移：功能点完成率71.43%，AI编码提效58.13%
- 营销资金设置端迁移：整体迁移时间降低62.73%

#### 日常迭代（低确定性）

**核心思路**：给恰好够用的精确知识，以查表替代推理

**关键机制**：
- **功能树**：将业务应用的功能点预先整理为树状结构，叶节点关联代码、接口、设计稿、Skill
- **分层研发指引**：功能点层（精确）+ 应用层（兜底）
- **D2C还原**：结构分析驱动实现策略（样式迁移 vs 结构重写）
- **API解析**：结构化解析 + 服务层mock
- **知识自动沉淀**：需求迭代产出回流功能树，形成提效飞轮

**落地效果**：
- 功能树覆盖需求点：生码完成度从40%~50%提升至80%~90%
- 部分需求类型采纳率达80%以上

## 核心方法论

1. **给恰好够用的精确知识**：上下文过载是生码质量下滑最常见的根因，知识的精确度比知识的完整度更重要。

2. **确定性逻辑交还工程，不确定性决策留给Agent**：将可枚举的路由、映射、格式处理收口到工程链路，让Agent专注于真正需要推理的创作任务。

3. **知识要形成正向循环**：通过功能树的自动沉淀机制，每次需求迭代的产出不只是代码，还会回流为下次生码的知识资产。
