# 多仓管理

> AI 全栈开发场景下，前后端（或多个服务）分布在独立仓库时的工程管理策略。核心挑战在于：让 AI 跨仓库理解代码关系，避免接口字段对不上的问题。

## 定义

多仓管理（Multi-Repo Management）是指在多仓库代码架构下，为 AI 编程工具提供统一代码理解能力的工程实践。传统微服务/前后端分离架构中，前端和后端代码通常分布在两个独立 Git 仓库中。在纯人工开发模式下，这不成问题——人类开发者可以同时打开两个项目。但在 AI 编程场景下，如果分开打开仓库，AI 生成后端接口时看不到前端的调用方式，生成前端代码时看不到后端的返回结构，接口字段对不上是家常便饭。

## 核心问题

### 多仓库带来的 AI 编程障碍

1. **上下文割裂**：AI 无法同时看到前端如何调用接口、后端如何返回数据
2. **字段对齐困难**：前端 VO 字段名与后端 JSON 返回字段名容易不一致
3. **重复描述成本**：开发者需要在提示词中分别描述前后端上下文，每次都要手动指定参考文件
4. **接口契约断裂**：前后端分别开发时，接口定义容易各说各话

## 主流解决方案

### 方案一：统一工作区（推荐）

将前后端代码放在同一个工作区目录下，让 AI 编程工具能同时索引两侧代码。

**Workspace 结构示例**：

```
workspace/
├── service-frontend/     # 前端仓库（独立 git）
├── service-backend/      # 后端仓库（独立 git）
├── sdd/                  # SDD 文档集中管理
│   ├── frontend/
│   │   ├── proposal.md
│   │   ├── spec.md
│   │   └── tasks.md
│   └── backend/
│       ├── proposal.md
│       ├── spec.md
│       └── tasks.md
├── .cursor/              # Cursor 配置
└── .claude/              # Claude Code 配置
```

**三大价值**：
- **[[Codebase-Indexing]] 覆盖两侧代码**：Cursor 等工作区级别的语义索引能对工作区内所有代码建立语义索引，AI 能跨仓库理解代码关系
- **上下文完整**：AI 同时能看到前后端代码，接口字段、命名风格自然对齐
- **SDD 文档集中管理**：前后端 SDD 文档在同一工作区，便于接口契约对齐和版本管理

**注意事项**：首次索引可能需要几分钟，索引完成前 AI 的跨仓库理解能力有限。<!-- confidence: EXTRACTED -->

### 方案二：显式引用与描述

不合并仓库，但在提示词中通过文件路径显式引用跨仓库文件。

- Cursor：利用 `@文件路径` 引用其他仓库的参考文件
- Claude Code：手动 `@文件路径`，但无法引用代码段，仅支持文件级引用

**局限**：依赖开发者手动维护引用关系，跨仓库越频繁、维护成本越高。

### 方案三：Mono-Repo 化

将多仓库合并为单一仓库。适合代码耦合度高、团队规模较小的场景。

**优势**：天然解决跨仓库理解问题，[[Codebase-Indexing]] 覆盖全部代码。
**代价**：失去独立仓库的独立 CI/CD、独立权限管理等好处，仓库膨胀后工具链性能下降。

## 与 AI 工具的关系

### Cursor 的多仓支持

Cursor 的 [[Codebase-Indexing]] 对工作区所有代码进行向量化嵌入，建立语义索引。当开发者提问时：
- AI 通过语义检索自动找到相关的 Controller、Service、前端组件
- 不需要手动指定每个参考文件
- 生成后端接口时能参考前端的调用方式，反之亦然

### Claude Code 的多仓处理

Claude Code 不支持内置的语义索引，跨仓库理解依赖模型自身能力：
- 仅支持 grep 检索
- 准确度严重依赖模型（如 Sonnet/Opus）的上下文理解
- 在长链路复杂任务中，需要手动管理 Subagent 分工

### 工具对比

| 维度 | Cursor | Claude Code |
|------|:---:|:---:|
| 多仓库默认工作区 | ✅ 同一 workspace | ✅ 同一工作目录 |
| 跨仓库语义检索 | ✅ Codebase Indexing | ❌ 仅 grep |
| 跨仓库文件引用 | 快捷键/拖拽即可引用 | 需手动 @文件路径 |
| 多 Agent 并行 | 多 Tab 即开即用 | 需手动注册 Subagent |

## 多 Agent 协作模式下的多仓管理

当采用多 Agent 并行开发时，多仓管理需要额外关注：

1. **SDD 先行**：在启动并行 Agent 前，确保前后端 SDD 文档中的接口契约已对齐
2. **接口契约即桥梁**：前端 Agent 依赖的接口定义 = 后端 Agent 实现的接口定义，需在 SDD 中严格对应
3. **分阶段验证**：前端先用 Mock 数据验证，后端独立编译验证，最后联调
4. **一个 Agent 一个职责**：每个 Agent 只负责一个仓库的代码生成，减少跨仓库上下文切换
