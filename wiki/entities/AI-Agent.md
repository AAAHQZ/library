AI Agent代表人工智能从被动响应到主动执行的范式转变。与传统的LLM仅根据当前输入生成响应不同，Agent能够：感知环境（通过工具调用获取外部信息）、维护内部状态（记忆和会话上下文）、制定行动计划（基于目标和当前状态推理下一步行动）、执行动作（调用工具、操作文件、发送消息）并从反馈中学习（自我纠错和知识更新）。当前主流的Agent架构包括单Agent系统和多Agent协作系统。多Agent场景下，多个具有不同专业能力的Agent通过通信协议协作完成复杂任务，这一范式在软件工程、客户服务、内容创作等领域已有初步应用。

## Agent 作为内容消费者

除了作为代码生成者和任务执行者，AI Agent 也正在成为开源基础设施的**主要内容消费者**（而非人类）。以 Pigsty.io 为例，92%+ 的流量来自 AI 爬虫和 Agent 类访问，传统 Web 分析工具已无法准确反映流量的真实构成。这意味着开源项目的文档、API、网站设计需要重新考虑"用户"的定义——不仅要服务人类读者，还要服务 Agent 消费者。

## 工程实践原则

### 人的并发瓶颈
人脑并发上限约 4-6 个 Agent 窗口，瓶颈不是能力不够，而是注意力有限。解决方案不是让 AI 替代人，而是让系统不再依赖人的实时在场。

### 决策层级
目标 → 代码 → CLI → Prompt → Agent。每往上一层，不确定性和成本增加一个量级。能在下层解决的，绝不上推。

### 脚手架 > 模型
模型升级：成本 +300%，效果 +20%；脚手架升级：成本 +50%，效果 +200%。优先投资脚手架（SDD、调度层、observability、constitution.md），而不是追最新最贵的模型。

### 从 Task-Driven 到 Goal-Driven
Task-Driven 解决执行问题（人派活，Agent 执行），Goal-Driven 解决迭代问题（Agent 围绕目标自主推进）。Goal-Driven 需要 5 个前提：目标清晰、边界清晰、状态可见、过程留痕、权限可控。
