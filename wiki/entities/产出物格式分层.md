# 产出物格式分层

> 按数据流向选格式的方法论：Web→LLM 用 Markdown（token 效率），LLM→Human 用 HTML（视觉密度+可交互），LLM→LLM 用 Markdown/JSON（可再塞回 prompt），Human→LLM 用 Markdown/纯文本。

## 简介

产出物格式分层是 Anthropic Claude Code 工程师 Thariq Shihipar 在 2026 年 5 月提出的方法论。核心主张：此前 AI 领域"all Markdown"的做法是把四种不同数据流向的格式需求混在一起凑合；系统成熟之后，每条链路应该选择最适合自己的格式。

## 四种流向与最优格式

| 方向 | 主语 | 受众 | 优胜格式 | 核心诉求 |
|------|------|------|----------|----------|
| Web → LLM | 网页内容 | 模型上下文 | Markdown | token 效率、语义保留、去除装饰噪声 |
| LLM → Human | Agent 产出 | 人类决策者 | HTML | 视觉密度、可分发、可交互 |
| LLM → LLM | Subagent 输出 | 父 agent | Markdown / JSON | 可被再次塞进 prompt、易解析 |
| Human → LLM | 用户输入 | 模型上下文 | Markdown / 纯文本 | 写起来快、便携 |

## 背景

2026 年 5 月，Claude Code 工程师 Thariq Shihipar 发表《Using Claude Code: The Unreasonable Effectiveness of HTML》，主张 agent 给人看的产出物应从 Markdown 切换到 HTML。同期 Cloudflare 推出 markdown.new，将 HTML 网页转为 Markdown 喂给 LLM，两件事看起来矛盾，实则是同一方法论的两个侧面——都在反对"Markdown 包打天下"的偷懒假设。

## 判断方法

一个简单的判断标准：这份产出物接下来会被喂回模型，还是给人做一次性决策？
- **喂回模型** → Markdown（可被再次塞进 prompt，语义信号清晰）
- **给人做决策** → HTML（视觉密度高、可交互、可分发）

## Markdown 仍然占优的场景

1. Agent 之间通信（Subagent → 父 Agent）
2. 配置和指令文件（CLAUDE.md、AGENTS.md、SKILL.md）
3. 版本控制和 long-form 文档（技术 RFC、长期维护的 wiki）
