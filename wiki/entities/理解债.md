# 理解债（Comprehension Debt）

> 一种由 Vibe Coding 带来的新型技术债：当开发者"感觉（Vibe）"代码是对的，但实际上并不理解背后的异步流转和数据状态时，这个应用从诞生的那一刻起就变成了无法维护的遗产。

## 定义

"理解债"是本文提出的概念，描述 Vibe Coding 范式下的一种独特风险：代码写了但我看不懂。当开发者完全依赖 AI 生成代码而没有进行人工 review 时，就形成了"理解债"——代码能跑，但没有人真正理解它的运行机制。

## 与传统技术债的区别

| 类型 | 技术债 | 理解债 |
|------|--------|--------|
| 产生 | 有意简化/时间压力 | AI 生成但未理解 |
| 修复 | 可通过重构解决 | 需要重新学习或重写 |
| 可见性 | 通常有记录 | 通常隐含在代码中 |
| 危险程度 | 可预估 | 难以预估——"感觉对"可能完全不对 |

## 核心问题

理解债的危险在于它的隐蔽性：
- **表面正确**：代码能跑，测试能过
- **深层危险**：AI 缺乏深层的"程序理论（Theory of the Program）"，无法保证复杂场景下的正确性
- **维护困境**：当问题出现时，开发者无法通过理解代码来定位问题

## Vibe Coding 墙

随着业务逻辑变得复杂，状态管理、多表关联、条件路由和第三方集成交织在一起，AI 的"上下文连贯性"开始断崖式下跌。传统的通用 LLM 在应对复杂企业应用时，往往会陷入"修好一个 Bug 引入两个回归（Regression）"的泥潭。

## 解决路径

1. **强制 Code Review**：如果一个应用 100% 由 AI 生成且没有人 review 过代码，那么这个应用就是企业的定时炸弹
2. **三层工作流**：确保每一个关键节点都有工程师在"Extend"层进行过逻辑审计
3. **平台层强制 RBAC**：让安全变成基础设施的一部分，而不是代码片段的一部分
