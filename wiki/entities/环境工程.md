# 环境工程（Environment Engineering）

> 面向 Agent 的环境工程范式，核心思想是解决 Agent 上下文供给问题——让 Agent 持续、低成本、可信地接入真实业务世界。由阿里云沈林在 2026 GenAICon 上提出。

## 简介

环境工程（Environment Engineering）是针对企业级 Agent 落地提出的新范式，其出发点是：Agent 落地的真正瓶颈不在模型能力，而在"上下文供给能力"——Agent 能不能持续、低成本、可信地接入真实业务世界。

## 关键信息

- **类型**：概念/范式
- **领域**：AI Agent 工程、企业级 AI
- **提出者**：沈林（阿里云智能集团高级技术专家）
- **相关概念**：[[EventHouse]]、[[上下文供给]]、[[统一Catalog]]、[[知识对账]]、[[Agent制品化发布]]、[[Context-Engineering]]

## 详细内容

### 五维框架

环境工程围绕五个核心维度展开：

| 维度 | 核心问题 | 解决方案 |
|------|----------|----------|
| 信息完备性 | Agent 能否看见真实业务世界？ | 主动监听、事件订阅、挂载查询 |
| 感官管理 | 信息能否被正确消费？ | 统一 Catalog——提前记录数据语义、Schema、新鲜度等 |
| 知识对账 | Agent 是否真正理解数据？ | Knowledge Wiki + 人机可审查的知识对齐机制 |
| 变更治理 | 知识更新是否可管理？ | 制品化发布——可回归、可灰度、可回滚 |
| 普惠门槛 | 接入门槛是否足够低？ | Serverless、按量付费、零运维 |

### 与 Context-Engineering 的关系

环境工程是上下文工程（Context Engineering）的扩展和升级：
- 上下文工程侧重于**上下文的组织、管理与注入方式**
- 环境工程从更宏观的**环境感知与接入体系**出发，涵盖信息采集、语义管理、知识对齐、变更治理和平台化交付

### 核心洞察

1. **AI Coding 跑得快的原因**不是代码更适合 AI，而是软件工程环境本身已经高度数字化
2. **看不见，就很难判断对**——信息完备性是 Agent 决策能力的前提
3. **信息不是越多越好**——Agent 需要的是"图书馆馆藏目录"而非整座图书馆
4. **知识不是信息囤积**——基于 DIKW 模型，知识需要可审查、可解释、可复用
5. **每一次知识迭代都是生产级变更**——需要完整的变更治理机制
