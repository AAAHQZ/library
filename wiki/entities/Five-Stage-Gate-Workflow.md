# Five-Stage-Gate-Workflow

> QQ音乐Harness Engineering的五阶段主流程 + 四道强制门禁机制

## 简介

Five-Stage-Gate-Workflow（五阶段四门禁）是Harness Engineering的流程骨架，旨在解决传统SDLC在AI协作语境里被随意跳过的问题。

核心原则：**错误越早被拦住，代价越低**

## 五阶段

```
阶段1: 初始化 → 阶段2⭐需求定义 → 阶段3⭐设计 → 阶段4⭐⭐开发 → 阶段5交付
```

| 阶段 | 内容 | 门禁 |
|------|------|------|
| 阶段1 | 初始化（目录骨架） | 无 |
| 阶段2 | 需求定义（撰写→评审） | ⭐需求评审门禁 |
| 阶段3 | 设计（预研→设计→评审+追溯） | ⭐设计门禁 |
| 阶段4 | 开发（任务拆分→Dev门禁→服务仓库检查→编码循环） | ⭐⭐两道门禁 |
| 阶段5 | 交付（测试验收→收尾） | 无 |

## 四道强制门禁

| 门禁 | 位置 | 阻塞条件 | 拦住什么 |
|------|------|----------|----------|
| 需求评审门禁 | 阶段2.2 | 需求文档不合格/评审未通过 | "需求没理解对" |
| 设计门禁 | 阶段3.3 | 设计评审未通过/追溯链不达标 | "方案漏了关键约束/没追溯到需求" |
| Dev进入门禁 | 阶段4.2 | tasks/features.json缺失或不合法 | "feature拆分不合规，开发起点不对" |
| 服务仓库检查 | 阶段4.3 | 三仓分支不一致/服务仓库未就位 | "三仓分支漂移/IDL契约仓未就位" |

## 设计原则

1. **门禁尽量少、尽量靠前** - 加多了研发绕开，加少了错误漏出
2. **门禁是"机读"的，不是"口头的"** - 每个门禁都有对应的Agent/Skill和markdown检查规范
3. **单一真相源** - 门禁口径收拢在`context/harness-framework/main-process-numbering.md`

## 错误代价递增曲线

- 阶段2改几行文档
- 阶段3改设计文档
- 阶段4.1改任务拆分
- 阶段4.3改环境/重切分支
- 阶段4.4代码/IDL已写、数据迁移已做 → 回滚代价极高
