# Vibe Coding

> 一种通过自然语言与AI交互、快速生成代码的编程方式，本质上是中年男人的"钓鱼"——一个合法且体面的独处避难所。

## 定义

Vibe Coding 是指用户通过自然语言描述需求，由 AI 自动生成代码的编程方式。用户无需理解代码如何生成，只需说出"帮我做个能查天气的小工具"，AI 便会完成从代码生成到运行的全过程。

## 核心特征

- **低门槛**：无需编程基础，只需知道自己想要什么
- **过程导向**：在乎"我说了算"的创造体验，而非最终产品
- **即时满足**：看着代码快速滚动、项目神奇地运行起来，获得类似钓鱼时鱼竿猛沉的快感
- **深夜仪式**：往往是凌晨时分，老婆孩子睡下后的独处时光

## 与钓鱼的类比

| 维度 | 钓鱼 | Vibe Coding |
|------|------|-------------|
| 工具 | 鱼竿 | Prompt（提示词） |
| 目标 | 鱼是否上钩不重要 | 代码能否上线不重要 |
| 本质 | 避难所/独处借口 | 避难所/独处借口 |
| 收获 | 过程里的"我说了算" | 过程里的"我居然还能做出点什么" |

## 为什么对中年人尤其宝贵

1. **身份夹缝**：白天是部门经理，晚上是丈夫和爸爸，唯独不是自己
2. **稀缺体验**：现实生活中最缺乏的是自我主宰感
3. **门槛极低**：AI 把准备工作踩到地板上，直接抵达创造最激动人心的部分
4. **失而复得**：年轻时从零到一的成就感被生活消磨，在某个平凡深夜被重新击中

## 典型场景

> "凌晨时分在社交媒体上最兴奋地晒出自己用 AI 做出的小产品的人，往往不是 00 后的年轻程序员，而是三四十岁的中年人。他们分享的并非自己的技术有多高深，而是一种失而复得的感动——'我居然还能做出点什么来'。"

## 系统性批判（来自高德团队实践）

高德大模型应用平台王树新在《告别"氛围编程"》中系统性地论证了 Vibe Coding 在团队级研发中的局限性：

### 三大问题

1. **自由发挥问题**：AI 生成的代码常天马行空，同一需求可能有三种实现方式，每种都"能跑"但都与现有架构格格不入。根因：业务理解不足、规范缺失。

2. **效率降低问题**：指令不够清晰时，在多轮对话中反复拉扯（"改一下"→改了→"不对是这样"→又改），来回几次不如自己写。

3. **关键信息丢失问题**：多轮对话中 AI 会"忘记"之前的重要约束。任务粒度过大时，开头说的架构要求到后面就消失了。

### 存量应用的三重风险

1. **研发是全链路过程**：编码只占整个链路约 30%。把编码环节优化 50%，整体提效只有 15%。AI 生成的代码可能带来更多 Code Review、调试和返工时间。

2. **存量应用 Vibe Coding 风险极高**：存量应用有历史包袱、隐式依赖、业务知识沉淀。AI "氛围编程"可能生成看起来完美但和现有系统完全不兼容的方案，问题可能在上线后才暴露。案例：AI 修改核心接口参数顺序，单元测试全过，上线后导致三个下游服务报错。

3. **大型需求超出单次 AI 对话能力边界**：涉及前后端十几个模块的重构任务无法在一个对话里完成，AI 上下文窗口有限，注意力会分散。

### 结论

AI 编程要从"个人技能"升级为"团队级工程能力"，从"氛围编程"进化为"规范驱动、工程治理"的研发范式。**Vibe Coding 适用于个人项目、新项目、小脚本，但不适用于团队协作、存量应用、大型需求。**

## 控制论视角

Vibe Coding 本质上是"以少御多"的尝试——用户用自然语言（有限多样性）描述复杂需求（无限多样性），AI 尝试以此为输入生成代码并维持系统稳定。[[Ashby-必要多样性定律]]（Only variety can destroy variety）预言了这条路的瓶颈：除非大幅降低问题的多样性（如通过 [[Spec驱动开发]] 规范化输入），否则失控是结构性问题而非工程问题。
