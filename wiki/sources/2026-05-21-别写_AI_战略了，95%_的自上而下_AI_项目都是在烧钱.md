# 别写 AI 战略了，95% 的自上而下 AI 项目都是在烧钱

## 核心观点

### 1. Claude Code 的诞生：Bottom-up 的典型案例

Boris Cherny 2024年9月加入 Anthropic 后，因为"不想写 UI"在终端里直接和 Claude 对话，最初版本极其简陋。他在公司内部论坛发帖介绍原型，收到**两个赞**。后来 Claude Code 上线第一天 20% 工程师用，第五天 50%。目前占 GitHub 公开代码提交量的 **4%**（2026.02 SemiAnalysis 数据）。

> "那些最酷、最有影响力的 AI 产品，几乎都不是在会议室里规划出来的。它们诞生于一种更混沌、更有机的方式——'自下而上'。"
> — Mike Krieger, Anthropic CPO

### 2. Product Overhang（产品悬垂）与 Latent Demand（潜在需求）

**Product Overhang**：模型已经具备某种能力，但还没有产品去承接它。Claude 早就能操作文件系统、写脚本、做分析，但在 Claude Code 出现之前，没有人给它一个「入口」去做这些事。

**Latent Demand**：40% 的 Facebook Marketplace 帖子是买卖东西，但 Facebook 从没设计过这个功能，是用户自己发明的用法。Claude Code 也是——非技术人员在终端里做报销、分析基因组、恢复老照片，没人设计过这些场景。

**核心方法论**：不要只观察用户在做什么，也要观察模型试图做什么。两种潜在需求叠加在一起，往往指向最有价值的产品方向。

### 3. Top-down 在 0→1 阶段失效的原因

- MIT 2025 年商业 AI 状况报告：**95% 的 GenAI pilot 失败**
- 成功的 5% 几乎都有一个共同特征：不是中心化 AI 卓越中心（CoE）推的，是一线管理者和「超级用户」自己发现需求、自己选工具、自己跑通了，然后被组织承接
- 看起来像 top-down 的胜利，本质是 **bottom-up 先跑通了被组织追认了**
- OpenAI 自己的案例：Codex 本来是最早的 AI 编程产品，战略重心转移后团队被拆散，给了 Anthropic 一个窗口期让 Claude Code 从 side project 长成行业标杆

### 4. 场景区分：0→1 vs 1→100

| 阶段 | 特征 | 有效策略 |
|------|------|----------|
| **0→1 产品创新** | 模型能力边界不清楚、用户需求没被验证、正确形态未知 | Bottom-up 探索 |
| **1→100 规模化落地** | 产品形态已验证、需要组织保障和流程支持 | Top-down 推进 |

在 0→1 阶段，你能规划的只有「允许探索」和「准备承接」，而不是规划出一个产品路线图。

### 5. 0→1 阶段真正需要的 AI 工具哲学

回看 Claude Code 的设计哲学：
- 给模型工具，让它自主运行程序，不强迫它以特定方式行事
- 观察模型的行为，找到它「试图做但还没被产品化」的事
- 降低个人验证一个想法的成本——从想到到验证，周期越短越好
- 不预设用法场景，让用户和模型共同涌现使用方式

### 6. 判断框架：帮人探索 vs 帮人执行

| 类型 | 特征 |
|------|------|
| **帮人探索的工具** | 降低验证成本、不预设正确答案、让试错变便宜、用户不清楚自己要什么也能用 |
| **帮人执行的工具** | 提高确定性任务的效率、优化已知流程、标准化输出质量 |

前者需要从个人使用中长出来，后者可以被自上而下设计。都叫 AI 工具，但其实是两个东西。
