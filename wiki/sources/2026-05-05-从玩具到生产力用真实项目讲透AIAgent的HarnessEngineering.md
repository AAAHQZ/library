# 从玩具到生产力：用真实项目讲透 AI Agent 的 Harness Engineering

> Harness Engineering 是将大模型纳入企业工程体系的核心方法，通过控制面设计把"聪明但非确定性"的模型变成可预测的协作者。

## 核心观点

### Harness vs 传统软件工程

传统软件工程管理的是「确定性」（代码bug是确定的），而 Harness Engineering 管理的是「非确定性」（大模型输出是概率性的）。Harness 不是某条提示词或工具，而是一整套控制面：提供唯一真相源、约束执行边界、接入业务能力、观测运行状态、让产出可验证可回归。

### 架构坐标系

用两个维度界定 Harness 边界：
- **X轴（执行流路由）**：静态预设 vs. 动态自主
- **Y轴（状态与上下文）**：隐式内部 vs. 显式外部

四个象限：无状态链（纯函数）、提示词驱动（AutoGPT类）、传统管道（LangChain类）、**Harness Engineering**（外部状态管理+模型意图）。

### 伪 Harness 与劣质 Harness

- **伪 Harness**：在 Prompt 里写大量 DO NOT（口头嘱咐），或一股脑塞 20 个 API（无边界约束）
- **劣质 Harness**：暴力死循环重试（盲打），强制瀑布流重型文档（官僚主义）

### 好的 Harness 三要素

1. **前置验证（Evaluator 沙盒）**：单测失败时在沙盒里基于证据触发 Retry
2. **最小真相源（Spec is Truth）**：维护轻量状态机文档，跨天无损恢复上下文
3. **物理门禁（Checkpoint Before Execute）**：模型破坏环境前必须拿到授权

### 工程师身份迁移

大模型时代，程序员核心价值从"亲手写代码"迁移到"定义目标、卡住边界、掌控节奏、验收结果"。不是完全放任模型，而是在关键架构、关键边界、关键异常上保持接管能力。

### 实施 SOP（8阶段）

目标收敛 → 状态恢复 → 上下文装配 → 任务分块 → 链路设计 → 执行前校准 → 外部验证 → 回写交接

每轮只给模型一个带边界的输入，必须先交付中间产物，用 Harness 控制点核对后才允许进入下一步。

### 三层目标盯防

1. **总核心目标**：整个项目要完成什么
2. **阶段性核心目标**：当前这几轮只收敛哪个主问题
3. **本轮动作目标**：这一轮只允许做哪 1-3 个动作

### 偏航信号识别

- 开始绕过阶段目标直接谈总目标
- 开始跳过中间产物直接声称要改代码
- 开始用主观语气替代客观证据
- 开始混淆阶段完成和全局完成

## 关键结论

**想释放大模型的生产力，第一步是先建好约束和轨道。** 程序员必须从"写代码的人"迁移成"控盘的人"，而 Harness 的作用就是让这种放权变得可控。
