# 云原生基础设施 + SRE 落地项目：从平台建设到稳定性工程闭环

|> 一套完整的云原生平台建设实践：通过 Kubernetes + GitOps + 可观测性 + SLI/SLO 闭环，实现从代码提交到生产稳定运行的统一工程体系。

本文涉及的云基础设施实践可参考 [[IaaS]] 和 [[PaaS]] 实体页面的定义与区分。

## 整体架构：五层结构

### 架构全景

```
开发者 → GitLab CI → Argo CD → Kubernetes Production
                ↓
        Observability Plane (Prometheus/Loki/Grafana)
                ↓
        Security Plane (RBAC/OIDC/Secret管理)
                ↓
        SRE Control Loop: SLI/SLO → Alert → Oncall → RCA
```

### 五大设计原则

1. **声明式优先** — Git管理所有配置，减少手工改线上
2. **控制面与数据面解耦** — GitLab构建、Argo CD交付、K8s调度、监控观测各司其职
3. **平台标准化优先于个体优化** — 统一模板：健康检查、资源限制、监控暴露、日志规范
4. **以用户感知为核心定义可靠性** — 围绕成功率、延迟、饱和度、错误预算构建监控
5. **为高并发和多团队协作预留扩展位** — 组件选型考虑跨环境发布、多区域部署

## 核心技术组件

### Kubernetes 平台

| 组件 | 选型 | 作用 |
|------|------|------|
| 容器运行时 | containerd | 轻量、稳定 |
| CNI | Calico | 三层路由 + NetworkPolicy 多租户隔离 |
| Ingress | Nginx Ingress / 云厂商LB | 南北向流量接入 |
| 存储 | Ceph/NFS/云盘CSI | 状态型数据持久化 |

### GitOps 持续交付

```
代码提交 → GitLab CI → 镜像构建签名 → 更新GitOps配置 → Argo CD自动同步
```

**Argo CD 优势**：`selfHeal` 机制确保即使手工改动线上，也会被拉回 Git 期望状态

### 可观测性体系

**三层监控**：
- 基础设施层：节点CPU/内存/磁盘/网络
- 平台中间件层：Nginx/MySQL/Redis/Kafka
- 业务服务层：请求量/成功率/P99延迟

**Prometheus + Loki + Grafana** 组合实现指标、日志、链路的统一观测

## SRE 核心实践

### SLI/SLO/Error Budget 闭环

以支付服务为例：
- **SLI**：`/api/payments` 接口 5 分钟窗口成功率
- **SLO**：月度成功率 ≥ 99.95%
- **Error Budget**：每月允许约 0.05% 失败

**意义**：发布节奏与错误预算挂钩，团队能客观讨论"是否在透支稳定性"

### 告警分级

| 级别 | 场景 | 响应要求 |
|------|------|----------|
| P1 | 用户核心路径故障 | 立即升级处理 |
| P2 | 核心服务明显退化 | 快速介入 |
| P3 | 潜在风险/容量临界 | 当天处理 |
| Info | 趋势提醒 | 用于治理 |

### 故障自愈机制

- **startupProbe**：应用是否完成启动
- **readinessProbe**：是否可以接流量
- **livenessProbe**：是否需要重启
- **优雅终止**：SIGTERM 处理、连接收尾、事务提交

## 多环境与多租户治理

### 资源隔离

```yaml
ResourceQuota: 限制命名空间总资源上限
LimitRange: 防止"裸奔Pod"
PriorityClass: 核心服务抢占优先级
NetworkPolicy: 跨服务访问限制
```

### 高可用设计

- Master 节点跨可用区部署
- etcd 独立部署保证磁盘性能
- topologySpreadConstraints 避免副本集中
- PodDisruptionBudget 保证维护期间最小可用副本

## 安全治理

- **身份认证**：OIDC 对接企业统一身份
- **镜像安全**：漏洞扫描 + 镜像签名准入
- **密钥管理**：Sealed Secrets / External Secrets 对接 Vault
- **运行时安全**：禁用特权容器、只读根文件系统、Seccomp/AppArmor

## 关键启示

1. **SRE 不是多配告警，而是先定义可靠性目标** — SLI/SLO 是所有稳定性工作的起点
2. **GitOps 解决的不只是发布问题** — 可审计、可回滚、可复现才是核心价值
3. **可观测性必须面向用户体验** — 不是看"机器忙不忙"，而是看"用户有没有受影响"
4. **平台标准化是规模化前提** — 早期投入模板化，后期多团队接入才能高效扩展
