# 网盘存量代码迁移实战：我们如何用三层架构管住 AI 的输出

> 原文链接：https://mp.weixin.qq.com/s/z2QWu5hE5TAZifqT4jYzVw

## 基本信息

- **来源：** 微信公众号原创（三石）
- **字数：** 4978
- **实践场景：** 网盘 Android 存量代码转 KMP 项目
- **核心方案：** Skill + SubAgent + Agent Team 三层架构
- **标签：** Skill、SubAgent、Agent Team、AI Coding、KMP

## 核心观点

### 1. 背景与挑战：AI 接入的真实问题

网盘 Android 存量代码转 KMP 过程中面临的三个规律性问题：

1. **同一类操作，不同对话结果不一样**：AI 每次都在重新理解任务，上下文略有变化，输出就漂移
2. **任务越复杂，后期幻觉越多**：迁移页面要加载几十个文件，到后期早期文件细节被压缩，AI 开始生成"不存在的方法"
3. **并行处理没有机制支撑**：UI、布局、业务逻辑本可以同时处理，但只有一个对话窗口，只能串行

### 2. 三层方案全貌

| 层级 | 解决 | 位置 |
|------|------|------|
| **执行层 · Skill** | 单点执行不稳定 | 所有具体任务的执行单元 |
| **调度层 · SubAgent** | 长链路上下文膨胀 | 转化阶段的串行步骤调度 |
| **协作层 · Agent Team** | 多类型任务串行效率低 | 提取阶段的并行分工 |

**选型判断依据**：任务之间有没有强依赖
- 有依赖、需要串行 → SubAgent
- 无强依赖、可以并行 → Agent Team
- 具体的执行动作 → Skill

### 3. Skill：解决"同一件事做十遍结果不一样"

**为什么输出不稳定？** AI 每次都在"重新理解"任务，上下文中表达方式略有差异，对任务的理解就不一样，输出自然不稳定。

**Skill 的本质**：把"这件事怎么做"固化下来，是无状态的、可复用的规范文件。

**最有效的约束：加 Checklist**
- 把每个步骤拆成可以逐项打勾的检查项
- AI 在多步骤任务里很容易跳步，有了 Checklist，遗漏率明显下降

**分层管理文件**：核心规则放主文件，边界情况和细节处理放 references 目录。Skill 文件太长，AI 注意力会分散，关键规则被稀释。

**关键决策：提取和校验拆开**
- extractor：生成初始提取包
- validator：独立校验，关注"还缺什么"而不是"提了什么"
- fixer：只处理 validator 指出的缺口，定向补充

三者视角天然不同，能独立发现遗漏，出问题时能直接定位是哪一层的问题。

### 4. SubAgent：上下文膨胀不是偶发问题，是必然的

**现象**：同一对话处理完整迁移流程，前期运行不错，后期开始出现奇怪错误——生成代码引用了不存在的方法，资源 ID 映射关系对不上。

**根因**：当加载的文件超过一定量，AI 对早期信息的记忆开始模糊，用"看起来合理"的内容来填充被压缩的细节——这就是幻觉，比输出明显错误更难处理。

**SubAgent 思路**：把长链路任务按步骤拆开，每个步骤交给独立 SubAgent 处理，各自有完全隔离的上下文，互不干扰。

**步骤之间怎么传信息？** 用 Agent-Memory：
- 每个 SubAgent 完成工作后，把关键信息提炼成结构化文件存下来
- 下一个 SubAgent 启动时读取这份文件，在自己独立上下文里继续推进
- Memory 文件不是对话记录的备份，而是"后续步骤需要什么"的提炼

**有效做法**：只提炼结构化的关键信息（资源映射表、模块结构配置、已确认的接口契约），可以精确消费，不会引入噪声。

### 5. Agent Team：四路并行，收益比预期大

**问题**：页面代码提取涉及四类文件（UI 组件、布局文件、业务逻辑、资源文件），四类文件提取逻辑完全不同但彼此没有强依赖，串行处理效率低。

**Agent Team 方式**：四个专业 Teammate 并行执行，各自独立上下文，只关注自己负责的那类文件。

**关键发现**：专注分工本身也会提升单类任务的处理质量。一个 Teammate 只盯一类文件，对该类型的依赖模式会建立更强的识别。

**跨 Teammate 依赖**：通过 Mailbox 消息通道实时处理，不需要等一个提取全部完成再补充，也不需要重新启动一轮。

**效果**：总耗时接近四类任务里最慢的那条路径（不是四段累加），专注度提升带来的提取质量改善比速度收益更明显。

### 6. 经验总结

1. **Skill 是被真实错误倒逼出来的，不是提前规划出来的**：反复纠错里逐步补充边界条件和反例，迭代过程本身就是价值
2. **上下文膨胀的临界点比想象中早，漂移比崩溃更危险**：幻觉是逐渐偏离，每一步看起来都还好，到最后才发现方向不对。及早拆分任务更划算
3. **Memory 文件要提炼，不要备份**：有效的是只传递"后续步骤需要消费的结构化信息"
4. **并行的收益不只在速度上**：分工带来的专注度提升，对任务质量的改善往往比缩短的时间更重要

## 关键概念

- **Skill**：把一类任务的执行方式写成可复用的规范文件，无状态，稳定输出
- **SubAgent**：解决长链路上下文膨胀，按步骤拆开各自独立上下文，步骤间通过 Memory 传递
- **Agent Team**：解决多类型任务并行效率低，多个专业 Teammate 并行执行，各司其职
- **Mailbox 消息通道**：Agent Team 跨 Teammate 协调机制
- **Agent-Memory**：SubAgent 间传递结构化关键信息的文件
