## 核心观点

### AI生码四大痛点

| 痛点 | 描述 |
|------|------|
| **写不对** | AI没有完全按照用户意图完成功能，轻则存在缺陷，重则无法运行 |
| **写不好** | 产出的代码不符合要求（代码质量/代码风格/实现方案） |
| **写不了** | 项目隐含逻辑太多、文件结构复杂、耦合度高，AI完全无法按预期完成任务 |
| **改不动** | 迭代场景中AI一直无法输出正确结果，在错误中循环，甚至改坏其他部分 |

### 五大成因与解法

1. **项目/需求隐含信息过多** → 使用明确声明的NPM包、Artifact文档工具、MCP工具/项目知识库
2. **用户输入不精准** → 提供prompt模板、Spec Coding方案、高频场景约定（AGENT.md）
3. **任务复杂度高** → 识别重复工作流进行封装复用、复杂任务拆分、固定实现思路减少思维负担
4. **缺少自检环节** → 代码质量自检（AI CR助手）+ 功能自检（MCP页面测试/单测）
5. **模型/Agent能力限制** → 上下文复用、严格约束文档、自检环节、By Case分析

### 核心方法论

- **最大化复用**：模块优先（黑盒设计）+ 胶水编程 + 工作流沉淀 + 文档先行
- **自然语言第一**：PRD即单测，文档即代码，优先修改文档而非代码
- **二八定律**：0%→80%（蜜月期）快速生成，80%→100%（深水区）需人工介入

### 全流程优化节点

```
前置准备 → 开发前 → 开发中 → 完成后
```

- **前置准备**：知识库、代码实现规范（AGENTS.md）、解耦的仓库结构
- **开发前**：明确需求内容、任务设计与拆分（组件拆分/流程拆分）
- **开发中**：过程文档、上下文窗口管理、自检机制
- **完成后**：问题分析、资产沉淀、基础规范迭代

### 两类实战场景

| 维度 | 需求驱动型 | 工程主导型 |
|------|-----------|-----------|
| **特点** | 强调"要什么功能"，AI自主决策 | 强调"怎么实现"，人工深度参与 |
| **代码规范度** | 低 | 高 |
| **验收标准** | 低 | 高 |
| **隐含知识量** | 低 | 高 |
| **AI角色** | 功能开发者 | 编码辅助员 |
| **案例** | 小二后台页面 | C端复杂业务 |

## 实践要点

1. **视图分离**：将视觉代码与逻辑代码分离，减少前端CR压力，提高复用度
2. **迭代型任务**：成功率低于新建型，因为AI天生重模仿而弱理解
3. **文档策略**：小而精优于大而全，根据AI实际偏离情况渐进补充
4. **版本管理**：复杂迭代时及时通过git保存版本（Commit分支）

## 原文精彩摘录

> 无论人工编码还是AI编码，最有效率的提效方案就是提高代码的复用度。

> AI的28定律：20%时间可以完成80%的任务，但剩下的20%要80%的时间。

> 认清AI在0%→80%（蜜月期）和80%→100%（深水区）的表现差异，重点是提高前80%的质量和后20%的效率。

> C端AI编码不好用的一个主要原因就是C端逻辑与视觉耦合度太高，而AI天生缺乏对视觉内容的感知力。
