# 从第一性原理思考 Agentic Engineering

> 一种工程师与 AI Agent 深度协作的系统性方法论，在保持质量标准的前提下提升研发效能

## 核心论点

Agentic Engineering 代表了一种截然不同的范式：AI 不仅是代码的执行者，也是问题分析、方案设计的思考伙伴；但最终的判断和决策权始终在工程师手中。它不是"让 AI 替你写代码"，而是将工程纪律与 AI 能力系统性地结合。

## 三条基本公理

### 公理 1：信息损耗（Information Loss in Software Engineering）

软件工程的本质是将人类模糊的意图逐步精确化为机器可执行的代码。这条链的每一步都可能引入信息损耗：

```
人类意图 → 自然语言需求 → 结构化设计 → 形式化代码 → 可执行程序
```

### 公理 2：LLM 的本质特征

LLM 是基于上下文进行概率性推理的系统，具有三个并列特征：
- **输出由上下文决定** — 对训练数据之外的知识掌握为零
- **输出是概率性的** — 同样的输入可能产生不同输出
- **工作记忆有限且易失** — 会话结束后没有跨会话的持久记忆

### 公理 3：人类认知是稀缺资源

工程师的注意力、判断力和决策能力是有限的，是整个人机协作系统的瓶颈。

## AI 带来的双面效应

| 维度 | 固有挑战 | AI 的改善 | AI 引入的新问题 |
|------|----------|-----------|----------------|
| 信息损耗 | 意图逐步失真 | 缩短反馈周期 | 概率性输出引入"似是而非"损耗 |
| 知识孤岛 | 隐性知识难以传承 | 通用知识即时可用 | 团队私有知识仍无法被 AI 利用 |
| 认知成本 | 理解复杂系统消耗大量认知 | 辅助代码解读 | 审查信息量暴增 |
| 重复性劳动 | 机械性工作占用大量时间 | 大幅自动化生成 | 生成成本骤降但验证成本未降 |

## 六条最佳实践

| ### 实践 1：Context Engineering

构建高信噪比的上下文供给系统。核心是按需注入而非全量灌入：
- **Spec-First** — 用结构化文档锚定意图（参见 [[Spec驱动开发]]）
- **Docs as Code** — 让文档成为上下文的一等公民
- **渐进式披露** — 按需展开，而非一次性灌入

### 实践 2：基于知识不对称的人机分工

用乔哈里窗模型分析知识状态：
- **开放区**（AI 知道 + 人类知道）— 极致自动化
- **盲区**（AI 不知道 + 人类知道）— 上下文注入
- **潜能区**（AI 知道 + 人类不知道）— 知识杠杆
- **未知区**（双方都不知道）— 协同探索

### 实践 3：AI 全链条参与

AI 从需求澄清阶段就开始参与，而非仅仅负责代码生成。不同阶段角色不同：
- 需求澄清阶段 → **引导者**（帮助工程师显式化意图）
- 方案设计阶段 → **协作者**（分析权衡、提出替代设计）
- 编码与测试阶段 → **执行者**（承担更大自主性）

### 实践 4：小任务推进、多层次验证

将大任务拆解为可独立验证的子任务，每步 AI 执行、人类校验后再进入下一步。验证层次：
- 意图层 → 需求评审、设计文档 Review
- 实现层 → Code Review
- 行为层 → 自动化测试
- 系统层 → 集成测试、性能测试

### 实践 5：Knowledge as Code

将团队共有知识编码为结构化的 Skill 和 Rules，存入仓库，版本化、可 Review、可迭代。

### 实践 6：Error-Driven Context Refinement

从错误中提取反馈闭环：犯错 → 诊断根因 → 检索现有知识 → 创建或更新 Rule/Skill → 预防复发。

## 结论

Agentic Engineering 的核心是：在约束条件下寻找最优可行解。不是最大化 AI 的产出量，而是最优化工程师认知带宽的分配。
