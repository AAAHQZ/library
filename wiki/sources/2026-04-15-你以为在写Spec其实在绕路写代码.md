## 核心观点

### 1. 足够精确的 Spec，本质上就是代码

Gabriella Gonzalez 在《A sufficiently detailed spec is code》中拆解了 OpenAI Symphony 的 SPEC.md，发现大量内容已经是伪代码、结构化算法和字段定义清单。要让 agent 稳定生成可运行实现，spec 必须写到接近代码的精度——这实际上是在用另一种语法编程。

Dijkstra 很早就指出：复杂系统协作依赖窄接口，口语化、宽松化表达不会减少劳动，只会把劳动转移成更多沟通成本和歧义。你不可能在降低表达精度的同时要求实现可靠性上升。

### 2. SDD 经常把团队带回瀑布流

STRRL 在《我对 Spec Driven Development 的看法》中指出：SpecKit 用久了之后，开发流程会变成「提前设计 + 脑内推演 + 一次性交付大块产出」，结果是巨大 PR 堆着不敢合。

软件不是一次性设计出来的，它是在持续变化里长出来的。需求会变、边界会变、约束会变、认知会变。而 SDD 在实践里最容易发生的事，是把「变化中的探索」提前冻结成「文档中的确定性」。最后开发节奏退化：先花大量时间写 spec，一次性生成大量代码，在后期集中爆雷，然后人工返工收拾残局——本质上是瀑布流的现代外观版。

### 3. Spec 的边界，就是系统的盲区

支持 SDD 的人常说「只要 spec 写得足够好，agent 就能稳定交付」。问题是工程里最难的部分，恰恰是你没预见到的部分。STRRL 直言：覆盖到的地方执行还行，覆盖不到的地方就是一坨大便。

Gabriella 实测按 Symphony 的 spec 让 Claude Code 生成 Haskell 版本，结果并不可靠，修了很多轮也有行为不符合预期的地方。YAML 这种规格已经非常细、还有一致性测试套件的标准，现实里仍然有大量实现不完全符合规范。

真正的问题不是「有没有 spec」，而是：你是否承认 spec 永远不可能完备，是否为「spec 之外」预留了快速反馈和修正机制，是否把工程质量建立在可验证循环上而不是文档完整性的幻觉上。

### 4. 当 Spec 也开始追求速度，它会退化成「文档形状的 Slop」

在「快速交付」压力下，spec 经常变成另一种流水线产物：句子很多、术语很全、结构看起来很专业，但整体缺乏真正的设计判断和一致性。Gabriella 批评 Symphony 某些章节像「specification-shaped sentences」——像规格说明，但不是规格说明该有的思考密度。

如果先用 AI 快速生成一份「像样的 spec」，再让 AI 按这份 spec 生成代码，中间其实是两次信息压缩和两次语义漂移。每一层看着都在「提炼」，但总信息量很可能在流失。

## 务实做法

作者现在的做法：

- 重要系统照样写设计文档，但只写**关键决策和关键约束**
- 不追求前置完整性，追求**小步可验证**
- 让 agent 快速产出，把质量建立在**测试、回归和审查闭环**上
- 文档服务于沟通和决策，不服务于「幻想一次性正确」

软件工程最贵的成本从来不是「敲代码」这件事，而是理解问题、做出取舍、承担后果。这部分，今天没有被 SDD 解决。
