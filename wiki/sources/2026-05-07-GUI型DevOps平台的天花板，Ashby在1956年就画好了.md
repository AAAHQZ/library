# GUI 型 DevOps 平台的天花板，Ashby 在 1956 年就画好了

## 核心论点

GUI 型 DevOps 平台（打开浏览器看到项目/环境/服务/部署/审批流的图形化产品）存在结构性天花板：产品团队在 UI 里预先定义核心概念，但工程组织的多样性远超产品团队的编码容量，这撞上了 Ashby 必要多样性定律。

## 关键概念

### 必要多样性定律（Law of Requisite Variety）

> **Only variety can destroy variety.**
> 控制器的多样性必须不小于扰动的多样性。

Ashby 用博弈模型证明：要控制系统，控制器的应对动作数必须至少等于扰动的种类数。`以少御多`注定失败。

### 工程组织的多样性

6 个维度（服务命名、环境模型、部署单元、变更类别、依赖关系来源、审批合规），每个维度 5 种取值 → 15,625 种形状。真实世界远不止 6 个维度。

### IaC 的出路：多样性外包

Terraform/Pulumi 把多样性外包出去：
- Cloud API 多样性 → provider
- 组织多样性 → module
- 部署模式多样性 → user code

多样性住在 user-space，核心代码量不爆炸。

### GUI 的困境：语义无法外包

GUI 这个介质天生没法外包语义——你没法给用户一张表单、让他在表单里定义新的表单。语义的注入面太窄。

GUI 型平台的慢性病：
- 历史遗留的"模式开关"没人敢删
- 新客户的概念和组织对不上
- if customer == X 只能由产品团队加

## 解决方案：语义层架构

GUI 的角色变了——从"告诉用户系统长什么样"变成"把基于用户语义算出来的结论展示给用户"。

**隐喻**：菜谱（写死做法）→ 语法书（定义语法，让用户自由表达）

- 什么是"服务""环境""变更" → 开放 schema，用户在 IaC/配置/注解中定义
- 审批、依赖、blast radius → 基于用户语义推理出来
- GUI → 推理结果的呈现层
