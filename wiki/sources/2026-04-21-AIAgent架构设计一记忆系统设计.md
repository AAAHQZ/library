# AI Agent 架构设计（一）：记忆系统设计

## 核心问题

语言模型本身没有状态，每次调用都从零开始。记忆系统要回答四个架构问题：

1. **存什么** — 哪些信息值得保留，哪些该丢弃
2. **存在哪** — 用什么介质，什么格式，什么生命周期
3. **怎么取** — 需要的时候怎么找到，精确匹配还是语义搜索
4. **怎么管** — 记忆怎么衰减、更新、压缩

## 记忆的四个层次

| 类型 | 说明 | 访问成本 | 生命周期 |
|------|------|---------|---------|
| 上下文记忆（In-context） | 当前 Token 窗口里的所有内容 | 最低 | 会话结束即消失 |
| 外部记忆（External） | 持久化在模型外部的存储（文件、数据库、向量库） | 中等 | 跨会话存活 |
| 情景记忆（Episodic） | 过去行为的结构化记录 | 较高 | Agent 从自身经验学习 |
| 参数记忆（Parametric） | 模型训练权重里编码的知识 | 无需检索 | 始终存在，无法运行时更新 |

|## OpenClaw：文件系统即记忆

### 核心设计决策

没有写进文件的，不存在。Agent 的所有长期状态必须持久化到磁盘上的 Markdown 文件里。

更多关于 OpenClaw 和 Hermes 的记忆系统设计对比，请参见 [[Hermes-Agent]] 实体页面。

```
~/.openclaw/workspace/
├── MEMORY.md          ← 长期记忆（精华）
├── SOUL.md            ← Agent 身份定义
└── memory/
    ├── 2026-04-12.md  ← 当日日志（短期）
    └── 2026-04-11.md  ← 昨日日志
```

### 两层记忆结构

- **短期层（memory/YYYY-MM-DD.md）**：当天工作日志，追加写入，不做整理
- **长期层（MEMORY.md）**：精华提炼，从日志中沉淀的稳定事实、用户偏好、工作规范

### 检索设计：混合搜索

语义搜索（向量相似度）+ BM25 关键词搜索并行运行，结果合并取最相关片段。

## 关键洞察

- **状态 ≠ 保存聊天记录**：真正的 durable state 是 agent 可以在冷启动后没有任何上下文历史的情况下读取、理解、续航的结构化工件
- **凡是不在 agent 运行时可见范围内的知识，就等于不存在**
