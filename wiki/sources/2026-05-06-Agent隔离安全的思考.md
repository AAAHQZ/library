# Agent 隔离安全的思考

> 批判 AI 行业对容器/微虚拟机隔离的过度投资，提出"策略优先于隔离"的安全模型。核心论点：Agent 不是攻击者，隔离不等于安全，绝大多数场景根本不需要 root 权限；真正的威胁是 Prompt 污染和工具间无隔离；Sandlock 方案基于 Landlock/seccomp 的进程级沙箱，无需特权守护进程。

## 核心观点

1. **Agent 不是攻击者**：容器/微 VM 的安全模型是为"不可信代码+主动攻击者"设计的，但 AI Agent 属于另一个场景
2. **隔离 ≠ 安全**：一个逃不出容器但能读取所有项目文件的 Agent，只是被关起来了，不是安全的
3. **策略 > 隔离**：拦截 `rm -rf /` 不需要硬件隔离，文件系统白名单足够；防止密钥泄露不需要虚拟机开销
4. **工具级安全缺失**：所有工具放同一沙箱 = 一个工具被攻破全部遭殃

## 当前行业的错误前提

**错误前提**：Agent 像攻击者一样主动突破隔离

**实际情况**：
- Agent 没有自己的意图，是按 prompt 执行的语言模型
- 真正威胁是 Prompt 污染：恶意输入诱导 Agent 执行 `curl` 窃取数据、`cat ~/.ssh/id_rsa` 读取密钥
- 这些操作在文件系统白名单和网络策略下完全可以管控，不需要硬件隔离

## Sandlock 的策略优先方案

**核心理念**：一切默认拒绝，按需显式授权

**内核接口**（全部非特权）：
| 接口 | 最低内核版本 | 功能 |
|------|------------|------|
| Landlock | Linux 6.12+ (ABI v6) | 文件系统访问控制、TCP 端口限制、IPC 和信号隔离 |
| seccomp-bpf | Linux 3.5+ | 系统调用过滤 |
| 用户命名空间 | Linux 3.8+ | 可选 UID 映射 |

**关键特性**：
- 不需要 root（内部和外部都不需要）
- 不需要构建镜像
- 启动时间 ~5ms（vs 容器 ~200ms，微 VM ~100ms）
- 无外部运行时，无特权守护进程
- 沙箱是进程自身的属性，而非额外运维的组件

**对比**：
| 特性 | Sandlock | 容器 | 微虚拟机 |
|------|----------|------|---------|
| 需要 root | 否 | 是 | 是 |
| 需要镜像 | 否 | 是 | 是 |
| 启动时间 | ~5ms | ~200ms | ~100ms |
| 文件系统隔离 | Landlock + COW | Overlay | 块级 |
| HTTP 级 ACL | 支持 | 无 | 无 |

## 工具级安全：每次工具调用单独沙箱

**容器方案的问题**：Shell、文件工具、浏览工具、数据库工具全部共享同一容器——一个被攻破全部遭殃

**Sandlock 的解法**：每个工具声明所需能力，调用时启动仅含该工具声明约束的新进程

- 浏览工具的沙箱：有网络但无法写盘
- 文件工具的沙箱：能写目录但无法联网
- 一个工具被攻破，攻击者无法获得其他工具权限

**这是最小权限原则应有的粒度**：不是按 Agent，不是按会话，而是按每一次工具调用

## XOA（Execute-Only Agent）模式

Sandlock 支持 XOA 模式：LLM 只生成代码，不接触不可信数据；生成代码在独立沙箱执行，输出通过内核管道直达用户，不会回流到 LLM 上下文

## 实际效果示例

**场景**：阻止 Agent 读取 `~/.ssh/id_rsa`

- **容器方案**：需要主动配置排除该路径，默认行为是 bind mount 中所有内容均可见
- **Sandlock**：Landlock 直接拒绝访问，因为该路径从未被授权
