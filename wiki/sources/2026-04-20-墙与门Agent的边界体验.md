# 墙与门：Agent 的边界体验

> Agent 的边界体验，是用户在不断试探中，建立「它能到哪里、会不会越界、越界时是否可信」的过程

关于 [[UX-设计]] 在 Agent 场景下的独特挑战，请参阅实体页面。

## 核心概念

本文讨论 Agent 时代的用户体验设计核心问题：**用户如何理解系统的行动边界**。

### 矛盾心智

传统软件有明确的界面边界（按钮、菜单、输入框），用户知道能做什么、不能做什么。但 Agent 只有自然语言输入框，用户的期待是模糊的：

- **抽象层面**：觉得 AI 好像什么都能做
- **具体场景**：沿用传统软件经验，默认没有入口的事情都做不了

### Wow Moment

Agent 的 wow moment 不是来自孤立功能，而是来自矛盾心智的缝隙：

> 用户以为这里有一道能力边界，结果 Agent 跨过去了，而且跨得自然、可控、有用。

## 边界体验的三个维度

用户关心的不只是「它能不能做」，还包括：

1. **能力边界**：能做什么、不能做什么
2. **责任边界**：什么时候主动行动、什么时候需要确认
3. **信任边界**：失败时是否可控、越界后是否可检查可回退

## 边界设计的四个原则

### 1. 让用户敢尝试

Agent 应主动暴露更进一步的行动可能。例如："我可以先帮你分析现有文件，再给出修改方案；如果你确认，我可以继续直接修改。"

### 2. 让请求变克制

区分风险层级，而非全部靠弹窗确认：

- **低风险**：直接完成（整理信息、生成草稿、读取上下文）
- **高风险**：明确说明并请求确认（删除文件、发送消息、不可逆操作）

### 3. 让惊喜被看见

Agent 做成用户没想到的事时，具体反馈关键：

> 它理解了什么、调用了什么、改变了什么、结果在哪里

wow moment 不是只靠能力发生，而是靠能力被用户感知到才发生。

### 4. 让失败可理解

失败时说明边界类型，帮助用户判断下一步：

- 权限边界：没有访问或操作权限
- 工具边界：当前没有可用工具
- 上下文边界：缺少必要信息
- 知识边界：无法确认事实
- 环境边界：运行环境不支持
- 规则边界：安全或合规限制

## 核心观点

好的 Agent 边界体验：
- 不是把能力清单摊开
- 不是让用户在混乱中盲目试探
- 而是设计**可探索的边界**

> **原来这里不是墙，而是一扇门。**
