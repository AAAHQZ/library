# 淘天营销中后台生码工作流最佳实践

> 淘天集团营销前台技术团队，系统总结了从本地 AI 辅助研发向云端托管生码收敛的最佳实践路径。核心方法：云端一体化研发 + 跨仓库工作区 + 可编排场景化工作流 + 功能树精准知识供给，实现 AI 生码采纳率 50-80%+。

## 核心方法论

1. **路径收敛**：从本地 Cursor/CodeAgent CLI 和云端 Alex 双轨收敛至云端托管生码（AoneSuper）
2. **Harness 思维**：给 AI 提供"模仿对象"而非自由发挥——找相似实现、复用优先、模仿着复制
3. **知识精确化**："给恰好够用的精确知识"而非"更多知识"，知识精确度 > 知识完整度
4. **确定性逻辑交工程，不确定性决策留给 Agent**：可枚举的路由/映射/格式处理收口到工程链路

## 为什么选择云端托管

**本地模式三大痛点**：
- 环境配置难以统一，问题排查协作成本高
- 生态用工的 AK（Access Key）管理困难，明文存储安全隐患
- 执行易中断，电脑息屏或网络断开导致任务失败

**云端 AoneSuper 优势**：借力集团已有 AI 基建，将投入重心从基建打磨转向业务效果优化

## 跨仓库工作区方案

**技术选型**：git submodule + Turborepo

**工作区结构**：
```
workspace/           # 需求工作区（独立 git 空间）
├── .claude/        # Agent 消费配置（Skills、MCP、SubAgent）
├── projects/       # 聚合所有需求相关子仓库（git submodule）
│   ├── frontend/   # 前端业务仓库
│   └── backend/    # 后端业务仓库
└── specs/          # 中间产物（需求理解、方案设计、任务列表）
```

**关键能力**：消除 git submodule 使用门槛，脚本自动化 submodule 操作（仓库初始化、代码拉取/提交）

## 可编排的场景化工作流

**整体架构**：CodeAgent CLI 的 Command 机制作为工作流节点推进方式

**节点类型**：
- **固定节点**：需求准备（入口）、构建部署（出口）
- **动态节点**：1~N 个，由需求场景定义，每个节点绑定独立 Command

**执行顺序**：需求准备 → 动态节点 → 构建部署

**典型场景示例**：
- ToB 需求迭代（商家营销工具）
- ToC 需求迭代（POP 弹窗）
- ToB 技术重构（营销资金设置端迁移）

## 两类研发任务的差异化策略

### 高确定性：迁移 / 重构
- **特点**：规则可穷举，旧写法固定、新写法固定，架构说明文档可覆盖全部场景
- **核心挑战**：规则传递失真（碎片化、歧义多）
- **解决思路**：
  1. 架构说明文档前置固化规则（新旧对照、禁止行为清单、接口映射）
  2. Skill 作为按需手册（表单开发 Skill/ProForm、表格开发 Skill/ProTable）
  3. 场景化工作流串联（了解旧架构→计划制定→编码实现→验证收尾）
- **落地效果**：功能点完成率 71.43%，TC 通过率 66.7%，AI 编码提效 58.13%

### 低确定性：日常迭代
- **特点**：需求方向无法提前穷举，改动范围跨功能模块
- **核心挑战**：信息过载 + 知识来源冲突 + 链路衰减
- **解决思路**：功能树实现"查表替代推理"
  - 预先梳理产品功能树：叶节点对应具体功能区域，绑定代码位置、接口定义、设计稿、Skill
  - 生码前通过 MCP 匹配功能树节点，获取精确代码位置和研发指引
  - 分层研发指引提上限、保下限；按需动态查询组件 API
- **落地效果**：功能树覆盖需求生码完成度从 40-50% 提升至 80-90%；平均采纳率 50%+，部分场景 80%+

## 知识沉淀正向循环

每次需求迭代产出 → 自动回流功能树 → 功能树覆盖越来越广 → 人工干预越来越少

**方法论沉淀**：
- "给恰好够用的精确知识，而非更多知识"
- "确定性逻辑交还工程，不确定性决策留给 Agent"
- "知识要形成正向循环，而非一次性投入"
