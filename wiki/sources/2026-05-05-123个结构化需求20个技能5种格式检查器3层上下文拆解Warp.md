# 123 个结构化需求、20 个技能、5 种格式检查器、3 层上下文：拆解 Warp 的 Harness 工程

> Warp 内部 60% 以上的 merged PR 由 Oz agent 生成。背后是一整套 Harness Engineering 实践：通过分层上下文结构、20 个结构化技能模块、五种格式检查器，将 agent 的行为空间约束在明确边界内，实现 124 万行 Rust 代码的稳定交付。

## 整体架构：三层上下文 + 两条验证链

### 三层上下文
| 层级 | 载体 | 解决的问题 |
|------|------|------------|
| 仓库级上下文 | WARP.md / CONTRIBUTING.md / FAQ.md | agent 理解项目架构、编码风格、协作流程 |
| 任务级上下文 | specs/ 目录下的 product.md / tech.md | agent 理解要实现什么、怎么实现、怎么验证 |
| 技能级上下文 | .agents/skills/ 目录下的 20 个 SKILL.md | agent 知道怎么执行特定类型的操作 |

### 两条验证链
- **静态检查链**：cargo fmt → cargo clippy → wgslfmt → clang-format → PSScriptAnalyzer → cargo-deny
- **动态验证链**：单元测试 → 集成测试 → Oz 自动 review → 人类 SME review

## Spec-Driven Development

关于 [[Spec驱动开发]] 的核心观点和争议，请参阅实体页面。

### 123 个结构化需求
specs/ 目录下每个子目录对应一个 feature 或 issue，包含：
- **product.md**：产品 spec，Behavior section 以编号的、可测试的不变量描述
- **tech.md**：技术 spec，锚定到具体代码位置，包含 Context（精确到文件和行号）

### 产品 Spec 设计哲学
**Behavior 是 spec 的全部，其余都是框架。** 必须覆盖：默认行为、状态转换、输入响应、边界情况（权限拒绝、离线、超时、竞态、多实例并发等）。

## 20 个结构化技能模块

覆盖完整软件生命周期：

| 阶段 | 技能 |
|------|------|
| 需求 | triage-issue-local、dedupe-issue-local |
| 设计 | write-product-spec、write-tech-spec、spec-driven-implementation |
| 实现 | implement-specs、add-feature-flag、fix-errors |
| 验证 | rust-unit-tests、warp-integration-test |
| 发布 | create-pr、review-pr、diagnose-ci-failures、promote-feature、remove-feature-flag |
| 维护 | resolve-merge-conflicts、update-skill |

### 技能约束性设计
每个技能不仅教 agent 怎么做，更重要的是**约束 agent 不能怎么做**：
- triage-issue-local：最多问 2 个 follow-up 问题
- review-pr：输出写入 review.json，禁止直接调用 GitHub API
- warp-ui-guidelines：禁止自创 button theme、硬编码颜色

## 约束工程：多层防护

### 编译时约束（.clippy.toml）
- 禁用 std::dbg 宏
- 禁用 std::time::Instant（WASM 不支持）
- 禁用 std::process::Command（Windows 会弹窗）

### 依赖约束（deny.toml）
- 许可证白名单：只允许 13 种许可证
- 安全审计：追踪 RUSTSEC 漏洞
- 通配符禁止：不允许版本依赖使用 `*`

### 格式约束：五种格式检查器
1. cargo fmt — Rust
2. cargo clippy — Rust lint
3. run-clang-format.py — C/C++/Objective-C
4. wgslfmt — WGSL shader
5. PSScriptAnalyzer — PowerShell

## 测试验证体系

### 单元测试
- 测试放独立文件 `${filename}_tests.rs`
- 虚拟文件系统：virtual_fs::VirtualFS 脱离真实 IO
- Feature Flag 作用域控制：RAII guard 自动恢复

### 集成测试
- 启动真实 Warp 实例，创建隔离 HOME 目录
- PreconditionFailed 机制：环境无效时返回特定错误，外层重试最多 10 次

## 核心启示

1. **约束 > 指令**：告诉 agent "不要用 std::time::Instant" 并通过 clippy 强制执行，比"请注意跨平台兼容性"有效得多
2. **Spec 是 Agent 时代的"类型系统"**：约束 agent 的实现空间，而不是告诉它怎么实现
3. **Multi-Tool 约束比 Single-Tool 指令更可靠**：LLM 对 system prompt 的遵循会随上下文长度衰减，而工具链的检查不会
4. **上下文格式应该是 agent-agnostic**：不要为特定 agent 框架定制格式，用 markdown + YAML frontmatter
