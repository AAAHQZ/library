# 为什么我在团队大力推进 Harness Engineering 的同时，却不认为它就是未来

> 原文链接：https://mp.weixin.qq.com/s/gwFqOnPEDC8-3yKwpQfO8g

## 基本信息

- **来源：** 微信公众号原创（基于 Harness Engineering 实践的深度思考）
- **字数：** 6352
- **核心观点：** Harness Engineering 是让 AI 进入软件工程生产系统的最佳实践之一，但只是大模型能力尚未完全内化工程上下文之前的中间层，不是终局
- **标签：** Harness Engineering、AI-Agent、软件工程、中间层、工程实践

## 核心观点

### 1. 什么是软件工程里的 Harness Engineering

**定义**：围绕大模型构建的一套工程化支架，使模型能够在明确上下文、受控权限、标准流程、可验证结果和团队知识约束下，稳定参与软件开发活动。

**核心不是"让 AI 更聪明"，而是"让 AI 在一个更适合工作的系统里工作"。**

包含七大要素：
- **任务描述规范**：需求、约束、验收标准、边界条件如何被表达
- **上下文供给机制**：代码库、文档、架构约定、业务规则、历史决策如何被模型读取
- **工具调用接口**：搜索代码、运行测试、访问日志、读写文件、查文档、调用内部平台
- **工作流编排**：从 issue 到设计、编码、测试、review、发布的协作路径
- **质量验证体系**：单测、集成测试、lint、类型检查、静态扫描、回归用例、评测集
- **权限与安全边界**：哪些操作可自动执行，哪些必须人工批准，敏感信息如何隔离
- **组织知识沉淀**：团队风格、模块边界、常见坑、最佳实践如何被持续更新

### 2. 落地五层路径

| 层级 | 内容 | 收益 |
|------|------|------|
| 第一层 | 个人工具（Copilot、Claude Code 等） | 个人效率提升 |
| 第二层 | 团队规则（代码风格、commit 约定、review checklist） | 个人经验→团队资产 |
| 第三层 | 仓库上下文（Agent 可读文档：架构、模块边界、测试命令） | 消除上下文盲区 |
| 第四层 | 工具链接入（MCP 等协议，Agent 可调用工具） | 从问答→可执行 |
| 第五层 | AI-native SDLC（AI 参与完整软件生命周期） | 工程系统的一部分 |

### 3. Harness Engineering 的四大价值

1. **降低上下文切换成本**：开发者不必反复解释项目背景，AI 可通过稳定上下文获取
2. **提升交付一致性**：把团队标准固化到规则和验证链路，而非靠个人临场发挥
3. **扩大自动化边界**：把 AI 的半结构化能力接入传统确定性自动化，形成更完整的自动化范围
4. **形成团队级学习系统**：每次失败的 AI 任务沉淀为更好的文档、规则、测试、权限边界

### 4. 为什么它是当前最佳实践

核心矛盾：模型能力在快速增强，但软件工程的真实约束仍然分散在代码、文档、CI、人的经验、组织流程和历史包袱里。

Harness Engineering 的价值正是把这些分散约束组织成模型可以使用的工作环境。

### 5. 为什么它只是中间过渡阶段

**当前需要 Harness 的原因**：
- 模型还不能稳定理解团队偏好 → 需要写 rules
- 模型无法天然拥有组织的完整工程记忆 → 需要手工维护上下文
- 模型还不能原生、安全、可靠地完成所有工程动作 → 需要复杂工具编排
- 模型输出仍可能在局部正确、系统错误之间摇摆 → 需要大量验证链路

**未来模型会吞噬的内容**：
- **吞噬上下文工程**：未来模型的上下文窗口、长期记忆、代码理解能力会继续增强，更多知识会被模型以更自然的方式吸收
- **吞噬工具编排**：未来的 Agent 会更像工程操作系统上的智能进程，知道什么时候读代码、查文档、跑测试、请求权限、询问人类
- **吞噬初级质量判断**：未来模型会更擅长识别代码异味、接口破坏、测试不足、迁移风险、性能隐患和安全问题

**类比**：Harness Engineering 是从"人类主导的软件工程"走向"AI 深度参与的软件工程"之间的脚手架。脚手架很重要，没有脚手架高楼建不起来，但如果有一天高楼稳定成型，脚手架不会成为建筑的主体。

### 6. 软件工程未来的几种猜想

1. **需求变成可执行对象**：需求文档同时面向人和 Agent，包含业务目标、边界条件、验收测试、风险等级，Agent 可直接生成方案、代码、测试和发布计划
2. **代码库变成会自我解释的系统**：持续生成模块地图、依赖关系、风险区域、变更影响分析
3. **团队变成人和智能体组成的团队**：测试 Agent、迁移 Agent、文档 Agent、review Agent、性能 Agent、依赖治理 Agent
4. **架构能力重新升值**：写代码的边际成本下降，真正稀缺的是判断：什么不该做、边界放哪里、系统复杂度如何控制
5. **软件交付更接近连续演化**：围绕目标指标持续生成、验证、发布和回滚，从"项目制"转向"演化制"

### 7. 软件开发从业人员的准备

1. **把隐性知识显性化**：能写清楚模块边界、业务规则、约束条件、失败模式、测试策略——因为不仅给人看，也给 AI 看
2. **提升任务定义能力**：能把"做一个功能"拆成目标、非目标、输入输出、验收标准、风险点、回滚方案
3. **掌握验证思维**：不要只问"AI 能不能写出来"，要问"我如何知道它写对了"
4. **理解工具和协议**：MCP、Agent Runtime、CI/CD、权限系统、代码搜索等会成为 AI 进入工程系统的接口
5. **保留架构判断和产品判断**：识别真正的问题，拒绝错误的需求，控制复杂度，设计可演化的系统
6. **训练与 AI 协作的工作习惯**：把 AI 当高吞吐、可并行、需要约束、需要验证的工程协作者
7. **主动参与团队 Harness 建设**：不要等模型变得完美，真正的红利属于那些在模型还不完美时就已经学会重构工程系统的团队
