# Agent 隔离安全的思考：为什么我们可能做错了

> 整个 AI 行业正在用容器和微虚拟机为 Agent 构建隔离方案，但这解决的是错误的问题。Agent 不是攻击者，真正需要应对的是 prompt 污染和误操作，靠的是策略（policy）而非隔离（isolation）。

## 核心观点

1. **Agent 不是攻击者**：Agent 是按 prompt 执行的 LLM，没有自己的意图，不会主动突破沙箱
2. **隔离不等于安全**：沙箱保护的是"能否逃逸"，但安全需要关注"沙箱内部能接触到什么"
3. **策略优于隔离**：文件系统白名单、网络策略、Landlock/seccomp 比容器和微虚拟机更有效

## 四大核心论述

### 一、Agent 不是你的敌人

**传统安全模型的误用：**

容器和微虚拟机的安全模型是为"运行不可信代码、抵御主动攻击者"设计的（如多租户云计算）。但 AI Agent 不属于这个场景。

**真正需要关注的是 Prompt 污染：**

- 用户提交需求 → 平台组装 prompt → 发送给模型
- 检索回来的文档、工具返回结果、用户输入都可能携带恶意内容
- prompt 注入是**应用层问题，不是内核层问题**

**应对策略 > 隔离：**
- 拦截 `rm -rf /` → 文件系统白名单
- 防止密钥泄露 → 不把密钥放入沙箱
- 限制网络访问 → 策略指定可访问主机

**XOA 模式（Execute-Only Agent）：**

LLM 只负责生成代码，不接触不可信数据；生成的代码在独立沙箱执行，输出通过内核管道直达用户，不回流到 LLM 上下文。

### 二、隔离不代表安全

**一个反直觉的事实：**

即使把 Agent 放进 Firecracker 微虚拟机（完美隔离），它照样能读取你的 SSH 私钥。原因：你把它放进了沙箱，并通过环境变量或 git clone 需求授予了 ~/.ssh 访问权限。

| 概念 | 解决的问题 |
|------|-----------|
| 隔离 | 能否逃逸 |
| 安全 | 沙箱内部能接触到什么 |

**Sandlock 的思路：白名单而非隔离边界**

- 一切默认拒绝：路径、网络、能力全部默认不可访问
- 开发者按需显式授权
- Landlock 直接在路径级别拒绝访问（比容器的 bind mount 精确得多）

### 三、Agent 几乎不需要 root

**沙箱内部不需要 root：**

编程 Agent 的核心操作（读源码、写改动、跑测试、调 API）都不需要 root。容器普遍以 root 运行是因为"路径阻力最小"，但这引入了不必要风险。

**沙箱外部的特权才是真正的负担：**

- Docker 守护进程以 root 运行
- kubelet 以 root 运行
- Firecracker 需要访问 /dev/kvm

**Sandlock 两侧均无需特权：**

依赖 Linux 内核接口（全部非特权）：
- Landlock（Linux 6.12+）：文件系统访问控制、TCP 端口限制
- seccomp-bpf（Linux 3.5+）：系统调用过滤
- 用户命名空间（Linux 3.8+）：可选 UID 映射

### 四、一个沙箱装所有工具形同虚设

**传统容器化沙箱的问题：**

所有工具（shell、文件工具、浏览工具、数据库工具）共享同一个容器：
- 浏览工具被恶意网页利用？它拥有与 shell 工具完全相同的权限
- 代码运行时读取环境变量？数据库连接串就在其中

**工具越多，沙箱越宽松，安全边界越失去意义。**

**Sandlock 的做法：每次工具调用单独开设沙箱**

| 工具 | 沙箱权限 |
|------|---------|
| 浏览工具 | 有网络，但无法写盘 |
| 文件工具 | 能写目录，但无法联网 |
| 被攻破 | 无法获得其他工具的权限 |

**最小权限原则的真正粒度：不是按 Agent，不是按会话，而是按每一次工具调用。**

## 与容器/微虚拟机对比

关于 [[OpenClaw]] 中 Agent 隔离安全问题的具体事件和教训，请参阅实体页面。

| 特性 | Sandlock | 容器 | Firecracker |
|------|----------|------|-------------|
| 需要 root | 否 | 是 | 是 |
| 需要构建镜像 | 否 | 是 | 是 |
| 启动时间 | ~5ms | ~200ms | ~100ms |
| 文件系统隔离 | Landlock + seccomp COW | Overlay | 块级 |
| HTTP 级 ACL | 有 | 无 | 无 |

## 结论

> "不要为不需要的基础设施付费去防御不存在的威胁。真正有价值的不是隔离，而是策略。"
