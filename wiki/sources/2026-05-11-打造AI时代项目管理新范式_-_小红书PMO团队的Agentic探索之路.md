## 核心框架：AI 项目管理 4 次迭代

| 版本 | 时间 | 核心命题 | 标志能力 |
|------|------|----------|----------|
| 1.0 项目知识问答薯 | 25 Q1-Q2 | AI 能不能理解项目管理 | AI 项目顾问，探讨方法论，提供框架建议 |
| 2.0 Agent + 多渠道接入 | 25 Q3-Q4 | Agent 能不能动手干活 | Todo管理/催更/周报总结/IM群交互 |
| 3.0 基于 OpenClaw 的个人助理 | 25 Q4-26 Q1 | Agent 能不能更智能执行任务 | 跨会话、跨渠道、越用越懂你的 AI 助理 |
| 4.0 PMOBP Agent | 26 Q1 至今 | Agent 能不能支持项目组所有人 | 为每个项目组提供共享的多渠道动态上下文 |

## 核心观点

### 1. 两个关键判断

1. **AI 是一种新的生产力**：把 routine 工作（拉群、催更、总结、识别 todo、写纪要）真正接管掉，把人留给真正需要决策的地方——目标、阵型、资源。

2. **AI 会越来越聪明**：未来必将经历「人管项目 → 人驾驭 AI 管项目 → AI 主导、人打辅助 → AI 完全接管项目」四个阶段。当前处于「人驾驭 AI 管项目」阶段。

### 2. 设计原则（从 2.0 到 4.0 坚持不变）

- **原子 Agent 必须自我闭环**：执行期间不依赖 master Agent 再次介入，要么成功返回结果，要么明确失败返回原因
- **复合 Agent 通过原子组合解决复杂任务**：例如「项目周报总结」由「读周报 + 读需求状态 + 汇总输出」等原子组成
- **复合 Agent 不能互相嵌套**：嵌套容易产生死循环，严重消耗算力

### 3. 3.0 阶段三件大事

**第一件：把 OpenClaw 做成个人助理**
- 把项目管理能力打包成 Skill，上架到公司内部 Skill Hub
- 基于 OpenClaw 为每位 PMO 创建「项目管理个人分身」
- 探索集体喂养「团队助理」的模式

**第二件：长记忆四件套**

| 组件 | 功能 |
|------|------|
| UserProfile | 用户画像 4 维（偏好/行为/常问问题/关联项目） |
| SessionMessage | 对话历史三级缓存（DB → 应用层 → Agent 层） |
| KnowledgeItem | 知识库 5 类，支持全文 + 标签搜索 |
| ContextBuilder | 动态拼装上下文，自动压缩 |

**第三件：自建项目注册平台**
- 作为 Agent 上下文的唯一数据锚点
- 承载项目名/成员/文档空间/周会文档/IM群/需求工具等核心主数据
- > "项目主数据是 Agent 冷启的关键"

### 4. 4.0 阶段核心升级

**从「1 人 × 1 Session」升级到「1 项目 × N 人 × M Session」**

为每个项目组提供一套共享的多渠道动态项目上下文，每人独立的 Session 实例都从这套上下文里获取信息。

**PMOBP Agent 定位**：
> 首先是信息中枢，其次才是任务执行者。

架构基于 OpenClaw 构建，复用其 Gateway、Session 模型、Multi-Agent 编排、IM 插件、Skill 体系、稳定性机制；新增两层专属能力：项目上下文路由（识别用户在问哪个项目）和角色感知（按管理者/项目负责人/项目成员自动调整回答粒度）。

### 5. 7 条实战经验

1. 不要等各种设施齐了才开始——架构跟着场景走，不要反过来
2. 评测集是 Agent 的「单元测试」——帮助客观判断 Agent 能力边界
3. 原子 Agent 必须能自我闭环——从 2.0 坚持到现在依然有效
4. 记忆结构化是更智能的底座——不能简单用「存聊天记录」代替
5. 领域 Agent = 领域 Source of Truth——核心价值不是再造轮子
6. 项目主数据先于场景扩展——没有主数据锚点，再多场景都是数据孤岛
7. AI 时代人人都是 Builder——每个 PMO 应该是 Builder + PMO
