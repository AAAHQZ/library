一个内部帖子只收到两个赞的 side project，如今贡献了 GitHub 4% 的代码提交。这件事说明了什么？
一个只收到两个赞的帖子
2024 年 9 月，Boris Cherny 加入 Anthropic 的第一个月，做了一件挺随意的事：因为不想写 UI，他在终端里直接和 Claude 对话。
最初的版本极其简陋——不能读文件，不能跑命令，什么工程相关的事都做不了。只是能和电脑打个招呼。然后他给了模型一些文件系统操作的工具。
"I gave it some tools to interact with the filesystem... Suddenly, this agent was really interesting. Claude exploring the filesystem was mindblowing to me."
— Boris Cherny, Claude Code 创始工程师
Boris 在公司内部论坛发了个帖子介绍这个原型。收到两个赞。没人觉得一个终端聊天工具能成什么气候。
后来的事大家都知道了——Claude Code 上线第一天，20% 的 Anthropic 工程师在用；第五天，50%。CEO Dario Amodei 看到日活曲线垂直增长，专门问 Boris：「你是不是强制工程师使用了？」Boris 说没有。
4%Claude Code 目前占 GitHub 公开代码提交量的比例（2026.02，SemiAnalysis 数据）
一个「不想写 UI」催生的 side project，现在是全球代码贡献量最大的单一 AI 工具之一。没有任何产品路线图预见过它。
AI 产品是"长"出来的
这不是个例。Anthropic CPO Mike Krieger（Instagram 联合创始人）在最近的访谈里说了一句话：
"那些最酷、最有影响力的 AI 产品，几乎都不是在会议室里规划出来的。它们诞生于一种更混沌、更有机的方式——'自下而上'。"
— Mike Krieger, Anthropic CPO
为什么会这样？Boris 给了一个框架：Product Overhang——模型已经具备某种能力，但还没有产品去承接它。Claude 早就能操作文件系统、写脚本、做分析，但在 Claude Code 出现之前，没有人给它一个「入口」去做这些事。
另一个概念叫 Latent Demand——潜在需求。Boris 用 Facebook Marketplace 类比：40% 的群组帖子是买卖东西，但 Facebook 从没设计过这个功能，是用户自己发明的用法。Claude Code 也是——非技术人员开始在终端里做报销、分析基因组、恢复硬盘里的老照片。没人设计过这些场景。
"不要只观察用户在做什么，也要观察模型试图做什么。两种潜在需求叠加在一起，往往指向最有价值的产品方向。"
— Boris Cherny
这意味着什么？AI 产品的最佳设计方法不是写 PRD、画原型、做用研，而是给模型工具、观察行为、产品化涌现。Claude Code 团队从不写 PRD，从不写功能规划书，直接用代码做原型。Boris 入职后第一个 PR 还因为「手写代码」被拒了——被要求用 AI 工具重写。
这不是个别团队的偏好，这是一种产品方法论的转向：从「设计者意图」转向「模型能力和用户行为的交汇点」。
最好的 AI 产品不是在会议室里规划出来的 | 来源：Unsplash
自顶向下为什么在 0→1 阶段失效
MIT 2025 年的商业 AI 状况报告给出了一个扎心的数据：95% 的 GenAI pilot 失败。成功的 5% 几乎都有一个共同特征——不是中心化 AI 卓越中心（CoE）推的，是一线管理者和「超级用户」自己发现需求、自己选工具、自己跑通了，然后被组织承接。
换句话说：看起来像 top-down 的胜利，本质是 bottom-up 先跑通了被组织追认了。
更讽刺的案例是 OpenAI 自己。Codex 本来是最早的 AI 编程产品，但 ChatGPT 爆火之后，公司战略重心转移，Codex 团队被拆散、资源被重新分配。这是一个自上而下的战略决策——为了集中力量做「更重要的事」。结果呢？给了 Anthropic 一个窗口期，让 Claude Code 从一个 side project 长成了行业标杆。
这不是说 top-down 永远无效。但它的适用场景是「1→100 的规模化落地」，不是「0→1 的产品创新」。
场景区分
0→1 产品创新：模型能力边界不清楚、用户需求没被验证、正确形态未知 → bottom-up 探索有效
1→100 规模化落地：产品形态已验证、需要组织保障和流程支持 → top-down 推进有效
混淆这两个阶段是绝大多数「AI 战略」失败的根因。在 0→1 阶段，你能规划的只有「允许探索」和「准备承接」。而不是规划出一个产品路线图。
AI 工具应该服务谁
如果接受了「最好的 AI 产品是从个人探索中长出来的」这个前提，那 AI 工具建设的方向就需要重新审视。
目前市场上大量的 AI 工具在做什么？帮企业管理 AI 采用、帮团队协调 AI 流程、帮组织监控 AI 合规。这些都有价值——但它们服务的是 1→100 阶段。
0→1 阶段真正需要的 AI 工具长什么样？回看 Claude Code 的设计哲学：
● 给模型工具，让它自主运行程序，不强迫它以特定方式行事
● 观察模型的行为，找到它「试图做但还没被产品化」的事
● 降低个人验证一个想法的成本——从想到到验证，周期越短越好
● 不预设用法场景，让用户和模型共同涌现使用方式
Boris 说他现在写代码，AI 占比已经从 2025 年初的 20% 涨到 100%。不是说他不写了，而是每一行代码都有 AI 参与——提议、修改、验证、重构。个人生产力的边界被极大地拓展了。
这给 AI 工具建设者的启示是：你的工具应该让一个人能做到过去一个团队才能做到的事——不是通过自动化流程，而是通过降低探索的验证成本。

写在最后：你的工具是帮人探索，还是帮人执行？
做个简单的判断测试：
判断框架
帮人探索的工具：降低验证成本、不预设正确答案、让试错变便宜、用户不清楚自己要什么也能用
帮人执行的工具：提高确定性任务的效率、优化已知流程、标准化输出质量
前者需要从个人使用中长出来，后者可以被自上而下设计。都叫 AI 工具，但其实是两个东西。
Claude Code 之所以成功，是因为它最初就是一个「帮人探索」的东西——Boris 自己探索模型能力时顺手做的。不是有人规划了一个「AI 编程助手」的品类然后去填。
如果你正在做 AI 产品，或者正在组织里推动 AI 工具建设，不妨问自己一个问题：你是在帮人探索，还是在帮人执行？如果是前者，放弃路线图，找到你的 product overhang。如果是后者，写好 PRD 推就行了。
不是不能规划，而是在 0→1 阶段，规划的成功率接近于零。真正有效的组织动作，是创造允许 bottom-up 探索发生的土壤——然后准备好在它长出来的时候承接住它。
