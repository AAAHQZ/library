# Agent 隔离安全的思考：为什么我们可能做错了
> 原文链接：https://mp.weixin.qq.com/s/Cx6YOgLPZM_3RPs0PSi-0w

> 原文链接：https://mp.weixin.qq.com/s/ICxIj6ydpftSLwgjA9zZAg

## 核心观点

整个 AI 基础设施行业都在投入大量资源为 Agent 构建隔离方案：容器、微虚拟机、gVisor 等各类重量级运行时。但如果我们重新审视这个问题，就会发现这套安全模型从根本上就站不住脚。**Agent 不是攻击者，隔离不等于安全，绝大多数场景根本不需要 root 权限，而将所有工具放进同一个沙箱更是形同虚设。**

过去一年，主流 AI Agent 平台几乎都引入了沙箱机制，但模式如出一辙：用容器或微虚拟机封装，套上硬件隔离，然后对外宣称"安全"。但实际上，整个行业正在解决一个**错误的问题**。

---

## 一、Agent 不是你的敌人

容器和微虚拟机的安全模型，是为特定场景设计的：运行不可信的、可能具有恶意的代码，抵御的是正在主动尝试逃逸的攻击者。多租户云计算需要这种防护，共享代码平台运行用户提交的代码也需要。但 **AI Agent 不属于这个场景**。

### Agent 没有自己的意图

Agent 是一个按 prompt 执行的语言模型。它没有自己的意图，不会主动琢磨如何突破沙箱，不会在没有指示的情况下从内核接口中寻找 0-day 漏洞。它执行的代码，是它接收到的指令的直接和间接产物。

### 真正需要关注的是 Prompt 污染

真正需要关注的不是 Agent 本身是否有恶意，而是 **prompt 是否被污染**。

在生产环境中，Agent 的 prompt 通常由开发者编写，终端用户无法直接接触。用户提交一个需求（"重构这个函数"、"分析这批数据"），平台将系统指令、工具定义和上下文组装为 prompt 发送给模型。系统 prompt 本身与其他应用代码一样可信，但**上下文窗口中的内容就未必了**：检索回来的文档、工具返回的结果、用户的输入，都可能携带恶意内容。

### Prompt 注入是应用层问题，不是内核层问题

攻击者通过 prompt 注入能做的是什么？让 Agent 执行一条看似正常的命令：curl 窃取数据，rm 删除文件，cat ~/.ssh/id_rsa 读取密钥。即使攻击手法更精巧，让 Agent 下载并执行外部脚本，该脚本也只是以**普通非特权进程运行**。它不会去链式利用安全漏洞进行内核提权，而只是调用 open() 读文件、connect() 建连接、unlink() 删路径。这些操作，**文件系统白名单和网络策略完全可以管控**。

同样值得注意的是 Agent 自身的误操作。大模型可能因理解偏差删错目录，覆盖本应只读的配置文件，或执行从训练数据中幻觉出来的危险命令。这些不是攻击，没有攻击者，只是 Agent 判断失误。但造成的损失同样真实，防御思路也一致：**限定 Agent 可触及的范围，确保局部失误不会波及整个系统。**

### 应对思路：策略 > 隔离

归根结底：

- 拦截 `rm -rf /` 不需要硬件隔离，文件系统白名单就足够了
- 防止密钥泄露不需要虚拟机的开销，不把密钥放入沙箱即可
- 限制网络访问不需要独立内核，一条策略指定可访问的主机就足够了

**应对 prompt 注入和 Agent 失误，靠的是策略（policy），不是隔离（isolation）。** 按工具、按路径、按主机做细粒度访问控制，比堆叠硬件隔离有效得多，因为它作用于 Agent 实际工作的那一层。

### Execute-Only Agent (XOA) 模式

Sandlock 的沙箱流水线架构支持 XOA 模式：LLM 只负责生成代码，完全不接触不可信数据；生成的代码在独立的沙箱阶段执行，输出通过内核管道直达用户，不会回流到 LLM 的上下文中。这是从**架构层面**消除 prompt 注入的可能：不可信数据根本无法进入上下文窗口，不依赖过滤，不依赖指令层级，而是在结构上消除风险。

## 二、隔离不代表安全

这个观点可能会让做基础设施的工程师感到不适。

### 逃逸 vs. 接触

你可以将 Agent 部署到 Firecracker 微虚拟机中：专用内核、最小根文件系统、virtio 网络设备，jailer 进程丢弃所有 capability。从隔离的角度看，无可挑剔。Agent 运行在独立的 vCPU 上，拥有自己的页表，客户机内核即使被攻破也无法影响宿主机。

**但 Agent 照样能读取你的 SSH 私钥。**

原因很简单：你把它放进了沙箱。可能是通过环境变量传入的，可能是因为 Agent 需要执行 git clone 而授予了 ~/.ssh 的访问权限，也可能是 Agent 能够请求 169.254.169.254 获取 IAM 凭证。

- **隔离**解决的是"能否逃逸"的问题
- **安全**解决的是"在沙箱内部能接触到什么"的问题

容器和微虚拟机领域花了十年优化前者。但对 AI Agent 而言，**后者才是核心问题**。一个逃不出容器，但能读取项目目录中每个文件、每个环境变量、每个网络端点的 Agent，称不上安全，只是被关起来了。

### Sandlock 的设计思路：白名单，而非隔离边界

一切默认拒绝：路径、网络、能力，全部默认不可访问。开发者按需显式授权：源码目录可读、临时目录可写、LLM API 可访问。其余请求一律由内核层的 Landlock 和 seccomp 拦截，无需 hypervisor 介入。

**实际效果如何？** Sandlock 沙箱中的 Agent 无法读取 ~/.ssh/id_rsa，尽管 Agent 进程与该文件之间不存在任何虚拟机、容器或命名空间边界。Landlock 直接拒绝访问，因为这条路径从未被授权。相比之下，容器要实现同样的效果，需要主动配置排除该路径，而其默认行为是 bind mount 中的所有内容均可见。

Landlock 当然也可以在容器内使用，两者叠加防护更强。但现实中几乎没人这样做。大多数容器化的 Agent 沙箱在挂载项目目录时，一次就挂载了整棵目录树。Agent 需要访问文件才能工作，而 bind mount 的粒度太粗。Landlock 则精确到路径级别：可以赋予 `/src` 只读权限、`/src/output` 写权限，同时阻止对 `/src/.env` 的访问。

## 三、你的 Agent 几乎不需要 root

Root 权限问题有两面：沙箱内部和沙箱外部。整个行业在两面都走偏了。

### 内部不需要 root

编程 Agent 的核心操作无非是读源码、写改动、跑测试、调 API。这些操作都不需要 root，除非开发内核这样的底层项目。它不需要独立内核，不需要块设备、虚拟网卡或 cgroup。但容器化的沙箱普遍以 root 身份运行 Agent，因为这样路径阻力最小：包安装不会报权限错误，文件权限不会成为障碍，反正有容器兜底。**这实际上是在引入不必要的风险。** 大多数生产环境并未配置用户命名空间映射，容器内的 root 就是宿主机上的 UID 0。

### 外部的特权才是真正的负担

这一点很少被讨论。容器和微虚拟机需要沙箱**外部**的特权基础设施来搭建隔离环境：

- Docker 守护进程以 root 运行
- kubelet 以 root 运行
- 即便是号称 rootless 的 Podman 也需要管理员配置 /etc/subuid
- Firecracker 需要访问 /dev/kvm，jailer 以 root 启动

这些特权组件位于沙箱边界之外。一旦隔离被突破，攻击者的落点周围遍布 root 进程。**为建立隔离而引入的特权基础设施，恰恰在隔离失效时放大了爆炸半径。**

### Sandlock 两侧均无需特权

Sandlock 内部不要 root，外部也不要 root。它依赖的三个内核接口全部为非特权操作：

| 接口 | 最低内核版本 | 功能 |
|------|-------------|------|
| Landlock | Linux 6.12+ (ABI v6) | 文件系统访问控制、TCP 端口限制、IPC 和信号隔离 |
| seccomp-bpf | Linux 3.5+ | 系统调用过滤，设置 PR_SET_NO_NEW_PRIVS 后由进程自行施加 |
| 用户命名空间 | Linux 3.8+ | 可选的 UID 映射，非特权用户即可创建 |

**整个沙箱在进程内部建立，fork() 之后、exec() 之前完成。没有外部运行时，没有特权守护进程。沙箱是进程自身的属性，而非需要额外运维的基础设施组件。**

从攻击面来看，每增加一个特权组件就多一个风险点。Docker 守护进程历史上出现过多个权限提升 CVE。为"保护"Agent 而叠加的特权基础设施越多，系统整体的攻击面越大。而 Sandlock 的沙箱即使被突破，攻击者获得的也只是一个没有任何特殊能力的普通用户进程，没有可利用的守护进程，没有访问其它沙箱的权限（Sandlock 可以嵌套）。

## 四、一个沙箱装所有工具，形同虚设

还有一个更深层的架构问题：**所有工具共享同一个沙箱。**

典型的 Agent 配备多种工具：执行命令的 shell、读写文件的文件工具、抓取网页的浏览工具、运行查询的数据库工具、执行生成代码的运行时。每种工具的风险画像不同，所需权限不同，出问题时的爆炸半径也不同。

容器化的沙箱将这些工具全部置于同一个容器中：

- Shell 工具和浏览工具共享文件系统、网络和环境变量
- 浏览工具被恶意网页利用了？它拥有与 shell 工具完全相同的权限
- 代码运行时读取了环境变量？数据库连接串就在其中

**沙箱保护了宿主机不受 Agent 影响，但工具与工具之间毫无屏障。**

这不是疏忽，而是根本性的设计缺陷。Agent 层面的安全与工具层面的安全是两个不同的问题，需要不同粒度的控制：

- **Agent 级安全**：管控的是 Agent 整体可以接触的资源——哪些文件、哪些网络、多少系统资源
- **工具级安全**：管控的是每一次工具调用可以接触的资源——浏览工具应有网络访问但不能写磁盘，文件工具应能写入特定目录但不能联网，shell 工具应有受限的可执行范围且无法接触凭证

两者混在同一个沙箱中，策略就不得不取所有工具所需权限的**并集**。任何一个工具需要网络，所有工具都获得网络访问。任何一个工具需要写权限，所有工具都可以写入。**工具越多，沙箱越宽松，安全边界越失去意义。**

### Sandlock 的做法：每次工具调用单独开设沙箱

每个工具声明自己所需的能力：读哪些路径、写哪些路径、访问哪些主机。调用时 Sandlock 启动一个新进程，仅按该工具自身的声明施加约束：

- 浏览工具的沙箱有网络但无法写盘
- 文件工具的沙箱能写目录但无法联网
- 一个工具被攻破，攻击者无法获得其他工具的权限

**这才是最小权限原则应有的粒度。不是按 Agent，不是按会话，而是按每一次工具调用。**

容器要实现这一点，需要为每次调用启动一个新容器。即使是 gVisor 这样的轻量运行时，每次启动也需要约 100ms。而进程 fork 加 Landlock 约束在毫秒级即可完成，使得逐调用隔离在生产环境下切实可行。

## 不要为错误的安全模型买单

不是说容器和微虚拟机毫无价值。在多租户云平台中，租户之间互不信任，硬件隔离确实必要。对于完全不可信的代码在彻底隔绝的环境中执行，微虚拟机是合理的选择。

但**大多数 AI Agent 场景并非如此**。它们是：

- 一家公司运行的内部编程助手
- 一个团队搭建的自动化测试流水线
- 一家企业部署的文档分析 Agent

真正的威胁是 Agent 因为 README 中的一句话就执行了 `pip install` 恶意包，或者因为理解偏差删除了生产环境的配置文件。

应对这些威胁，需要的**不是更多隔离，而是更好的策略**：

- 默认全部拒绝
- 按路径开放白名单
- 按工具做限制
- 在内核层强制执行

> 不要为不需要的基础设施付费去防御不存在的威胁。为每次 Agent 调用启动一个微虚拟机不是纵深防御，而是用为对抗性多租户设计的安全模型去解决一个细粒度访问控制的问题。当实际的攻击（一次 prompt 注入让 Agent 携带你的凭证执行 curl）完全在沙箱已授予的权限范围内即可得逞时，虚拟机边界带来的边际安全收益趋近于零。**真正有价值的不是隔离，而是策略。** 再多的硬件隔离也弥补不了一个权限配置过于宽松的策略。

## Sandlock：策略优先的轻量 Agent 沙箱

Sandlock 基于 Apache 2.0 协议开源，使用 Rust 实现。单一二进制文件，无外部依赖，无守护进程，无 root 权限要求。在 Linux 6.12+ 系统上即可运行。

### 与容器和微虚拟机的对比

| 特性 | Sandlock | 容器 | 微虚拟机 (Firecracker) |
|------|----------|------|------------------------|
| 需要 root | 否 | 是* | 是 (KVM) |
| 需要构建镜像 | 否 | 是 | 是 |
| 启动时间 | ~5 ms | ~200 ms | ~100 ms |
| 内核 | 共享 | 共享 | 独立客户机 |
| 文件系统隔离 | Landlock + seccomp COW | Overlay | 块级 |
| 网络隔离 | Landlock + seccomp 通知 | 网络命名空间 | TAP 设备 |
| HTTP 级 ACL | 方法 + 主机 + 路径规则 | 无 | 无 |
| 系统调用过滤 | seccomp-bpf | seccomp | 无 |
| 资源限制 | seccomp 通知 + SIGSTOP | cgroup v2 | VM 配置 |

> *Rootless 容器存在，但需要用户命名空间支持和 /etc/subuid 配置。

### 快速上手

**命令行：**

```bash
# 只读系统库，可写 /tmp，仅允许连接指定 API
sandlock run -r /usr -r /lib -r /etc -w /tmp \
  --net-allow-host api.anthropic.com -- python3 agent.py

# HTTP 级 ACL：精确到方法和路径
sandlock run \
  --http-allow "POST api.openai.com/v1/chat/completions" \
  --http-deny "* */admin/*" \
  -r /usr -r /lib -r /etc -- python3 agent.py

# 资源限制 + 超时
sandlock run -m 512M -P 20 -t 30 -- ./compute.sh

# COW 文件系统：写操作被拦截，成功后统一提交
sandlock run --workdir /opt/project -r /usr -r /lib -- python3 task.py

# Dry-run：预览变更内容，不实际执行
sandlock run --dry-run --workdir . -w . -r /usr -r /lib -- make build
```

**Python API：**

```python
from sandlock import Sandbox, Policy

policy = Policy(
    fs_readable=["/usr", "/lib", "/etc"],
    fs_writable=["/tmp/sandbox"],
    net_allow_hosts=["api.anthropic.com"],
)

result = Sandbox(policy).run(["python3", "agent.py"])
```

Agent 可以读取系统库、写入临时目录、调用 LLM API。SSH 密钥、环境文件、凭证等未显式授权的资源一概无法访问。无需容器。

**安装：**

```bash
pip install sandlock
```
