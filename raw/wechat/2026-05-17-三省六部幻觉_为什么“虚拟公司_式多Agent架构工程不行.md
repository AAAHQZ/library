前言
如果你正在考虑把多个AI Agent分别命名为"产品经理"、"架构师"、"测试工程师"，让它们像公司部门一样传递文档、协作完成任务——请停下来。
这个模式看起来很直觉、很有道理，但工程上有根本性缺陷。更重要的是：Anthropic、OpenAI、Google三家厂商在构建自己的Agent系统时，没有一家采用这个模式。
什么是“三省六部”架构
这是社区里广泛流行的多Agent设计思路（CrewAI、MetaGPT等框架都在用），核心模式是：
把复杂任务拆解成若干职能
每个Agent扮演一个角色（领导者、会计师、工程师）
任务在Agent之间像流水线一样传递
其设计哲学也是简单易懂，就是分工明确、组织严密，很像唐朝的三省六部制度
但是其问题在于：它解决的是人类的瓶颈，不是AI的瓶颈。
为什么“三省六部”从根本上是错的
1. 人类需要分工的原因：
单个人注意力有限
人有专业壁垒，切换成本高
人与人之间需要沟通来协调
2.而LLM的特性完全不同
同一个模型既能写PRD又能写代码，没有职业边界
模型的瓶颈是推理深度和信息完整性，不是注意力广度
模型之间没有"文化"和"默契"来补偿信息损耗
给Agent贴上"产品经理"的标签，不会让它更专业——但会让它拒绝越界。 而最有价值的推理往往发生在边界上，而三省六部模式在系统层面封死了这个可能性。
三省六部模式还来一个致命问题——信息在流转中死亡
三省六部模式里，Agent A产出文档传给Agent B，这个过程：
❌ 传递的是结论，不是推理过程❌ 原始意图在衰减，隐含假设在丢失❌ 每次传递都在累积误差❌ 工作流越长，最终输出越"局部正确但整体漂移"
人类组织靠会议、文化、非正式沟通来补偿信息损耗，Agent之间没有这些机制，而Agent要实现这些机制反而十分需消耗token
大厂的agent设计
若你看过codex、claude code的agent设计，你会发现三家厂商不也"传文件"吗？有什么本质区别吗？
关键区别：谁在写、写给谁、怎么更新
| 三省六部模式 | 外部状态文件模式 |
|------------|----------------|
| 角色间的单向交接 | 同一任务的增量日志 |
| A写完交给B，B不再回头 | 执行主体在每个checkpoint追加记录 |
| 信息被压缩成结论，推理过程丢失 | 下一个session读取的是任务完整历史 |
| **接棒就是断点** | **写和读的是同一个角色，只是时间不同** |
这就好比，每次工作交接，你都对对上一班的工作一无所知，这种情况下，你本就无法很好完成工作，因此同步好外部状态，成了一个难点，我们一起来看看大厂是如何处理这个问题的
Anthropic解法：Context Engineering + 显式状态文件
claude-progress.txt：跨session的工作日志
Git history：作为状态锚点，记录每一个增量变化
Initializer Agent：只在第一个session运行，建立环境、写好runbook
关键洞察：推理链的连续性不靠模型"记住"，靠显式的外部状态来锚定，而子agent上：
一个lead agent分解任务、协调subagent
subagent并行探索不同方向
结果回流给lead agent综合
💡 关键区别：Anthropic的subagent是功能性并行（同时撒网捞鱼），不是三省六部的职能性分工（接力赛）。
OpenAI解法：Compaction + Skills + 结构化Spec文件
长任务原则：在任务开始时就为continuity做规划。
实践案例：
给agent一个spec文件（冻结目标，防止方向跑偏）
生成milestone-based plan
用runbook文件告诉agent如何操作（同时作为共享记忆和审计日志）
结果：GPT-5.3-Codex跑了约25小时不间断，完成了一个完整的设计工具，保持了全程连贯性。
核心机制如下：
Server-side compaction作为默认primitive
previous_response_id让model能在同一个thread里继续工作
Skills：可复用的、版本化的指令集（这是工具和操作规程，不是"角色"）
Google解法：1M上下文 + Context-driven Development
硬扩窗口策略：Gemini的1M token context是明确的差异化策略。但他们自己也承认这不够用，推出了Conductor扩展：
把项目意图从聊天窗口里移出来
放进代码库里的持久化Markdown文件
哲学："不依赖不稳定的聊天记录，依赖正式的spec和plan文件"
Gemini 3新机制：Thought Signatures
在长session里保存推理链的关键节点
防止"reasoning drift"（长context里逻辑前后不一致）
agent的设计哲学
原则1：推理链不能断，只能分叉再合并
多Agent的正确用法不是流水线
一个主agent持有完整意图
子调用是为了深挖某个子问题
结果回流给主agent，不是传给下一个agent
原则2：显式外部状态，不靠模型记住
progress.txt、git history、spec文件、数据库
形式不重要，原则是：推理链的关键节点必须外化到持久存储里
原则3：多Agent的价值是并行覆盖，不是分工
Anthropic Research系统结论：性能提升主要来自"花了更多token"
多Agent适合breadth-first类任务（需要同时探索多个独立方向）
不适合需要连续推理、深度依赖上下文的场景
原则4：验证Agent是否定者，不是接棒者
如果要用多Agent做质量控制
正确设计：让一个Agent专门找另一个Agent的问题
对抗性检验，不是流水线传递
原则5：工具是工具，不是角色
给Agent配什么工具（bash、文件读写、搜索、代码执行）远比给它贴什么标签重要
工具决定了Agent能做什么；角色标签只是约束它愿意做什么
最好的多Agent系统，不像公司。它更像一个思考者的多次草稿——同一个大脑，在不同维度上展开推理，最终合并成一个连贯的结论。
总结
三省六部是一个让人感觉良好但工程上代价高昂的错觉。
它的真正成本不是直接的失败，而是让你的系统在复杂度上升时，以一种难以诊断的方式退化——每个节点都"看起来在工作"，但整体在漂移。
而agent构建的在于外部环境连续、上下文压缩和提示词编写
